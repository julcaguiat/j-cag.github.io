Infrastructure Recovery & Troubleshooting

Playbook 1: SERVER & STORAGE FAILURE RECOVERY

Overview

Playbook 1 focuses on recovering failures affecting enterprise servers, virtual machines, storage resources, and critical infrastructure services deployed within the JCAG-LAB environment. Unlike deployment documentation, this playbook begins after a service interruption has occurred and guides the administrator through a structured recovery process to restore normal operations.

The procedures throughout this playbook simulate responsibilities commonly performed by Infrastructure Support Technicians, Data Center Technicians, Systems Administrators, and Network Operations Center (NOC) personnel. Each scenario emphasizes structured troubleshooting, operational discipline, service restoration, validation, and documentation rather than simply fixing individual technical problems.

Whenever practical, failures are intentionally reproduced within the isolated JCAG-LAB environment using controlled and reversible methods. Potentially destructive failures that could result in permanent data loss are documented as production reference procedures and are not intentionally induced.

Learning Objectives

  • Apply a structured troubleshooting methodology to enterprise server failures.
  • Diagnose operating system, virtualization, storage, and application service failures.
  • Recover Windows Server and Linux virtual machines using snapshots and backups.
  • Restore enterprise services while minimizing operational downtime.
  • Validate infrastructure health following service restoration.
  • Document recovery activities using repeatable operational procedures.

The recovery procedures in this playbook reference the production-style enterprise infrastructure deployed throughout the JCAG-LAB environment. Understanding the relationships between virtualization, storage, and enterprise services is essential before attempting any recovery activity.

Most recovery procedures begin by determining whether the issue affects an individual service, an entire virtual machine, the underlying storage layer, or the Proxmox virtualization platform itself.

Enterprise Storage Architecture

The JCAG-LAB environment separates workloads into three independent storage tiers. Understanding which tier hosts a workload is critical because the appropriate recovery procedure depends on where the virtual machine and backup data reside.

Tier Storage Purpose
Tier 0 local-lvm (SSD) Performance-critical virtual machines and enterprise services
Tier 1 local-lvm-hdd-01 General-purpose virtual machines and lab workloads
Tier 2 HDD-02 Backup Directory Storage VZDump backup repository used for disaster recovery
Important:
  • Snapshots are recovery checkpoints only.
  • Snapshots are not backups.
  • Tier 2 storage contains the authoritative recovery copies.
  • Catastrophic VM failures require restoration from Tier 2 backups.

PB1-S01 — Windows Server 2019 Fails to Boot

Scenario Overview

The Windows Server 2019 virtual machine hosts several core enterprise services within the JCAG-LAB environment, including Active Directory Domain Services (AD DS), DNS, DHCP, and SMB file sharing. A failure to boot renders these services unavailable, affecting authentication, name resolution, address assignment, and file access across the environment.

This scenario demonstrates a structured recovery process beginning with basic virtualization checks before progressing to guest operating system recovery. Because boot failures may result from virtualization issues, storage problems, or operating system corruption, troubleshooting must proceed methodically.

This scenario is classified as Safe Lab Validation. The failure can be intentionally reproduced using non-destructive methods and safely recovered using snapshots created during Playbook 0.

Business Impact

Affected Service Operational Impact
Active Directory Domain authentication unavailable
DNS Name resolution fails
DHCP Clients cannot obtain new leases
SMB Network shares unavailable
Group Policy Policy processing fails

Although existing client sessions may continue operating temporarily, new logins, DHCP renewals, and domain-dependent applications will begin to fail as services become unavailable.

Observable Symptoms

  • Virtual machine remains powered off.
  • Virtual machine immediately powers off after startup.
  • Windows enters Automatic Repair.
  • Black screen displayed after BIOS.
  • Blue Screen of Death (BSOD).
  • Proxmox reports the VM is running but no console is displayed.
  • Users cannot authenticate to the domain.
  • DNS and DHCP services become unavailable.

Safe Lab Validation Procedure

Objective

Safely simulate a boot failure without permanently damaging the operating system or virtual disk.

Method 1 — Stop the Virtual Machine

qm stop 101

Attempt to access Active Directory resources from a workstation.

Expected observations:

  • VM reports as stopped.
  • Remote Desktop unavailable.
  • Authentication eventually fails.
  • DNS queries begin failing.

Method 2 — Disconnect the Boot Disk (Safe)

Shutdown the VM before making hardware changes. 

qm stop 101

Within the Proxmox web interface:

  • Select VM 101.
  • Open Hardware.
  • Detach the primary boot disk (do not delete).
  • Start the VM.

The virtual machine will fail to locate a boot device.

Never delete the virtual disk. Only detach it temporarily for this validation. Reconnect the disk before completing the exercise.

Initial Assessment

Begin troubleshooting at the virtualization layer before assuming an operating system problem.

Step 1 — Verify VM Status

qm status 101

Example output:

status: running

Questions:

  • Is the VM running?
  • Is the VM stopped?
  • Did it unexpectedly shut down?

Step 2 — Verify VM Configuration

qm config 101

Verify:

  • Boot order
  • CPU allocation
  • Memory allocation
  • Primary disk attachment
  • Network adapter

Diagnostic Procedure

Step 1 — Review Recent Tasks

journalctl -xe

pvesh get /nodes/proxmox/tasks

Look for:

  • Failed startups
  • Storage errors
  • Lock errors
  • Snapshot failures

Step 2 — Verify Storage Availability

pvesm status

Example output:

Name                 Type      Status
local-lvm            lvmthin   active
local-lvm-hdd-01     lvmthin   active
backup-storage       dir       active

Questions:

  • Is Tier 0 available?
  • Is the VM disk visible?
  • Has storage become read-only?

Step 3 — Verify Disk Attachment

qm config 101 | grep scsi

Expected output:

scsi0: local-lvm:vm-101-disk-0

If no disk is listed, the VM cannot boot because no boot device is attached.

Root Cause Analysis

Possible Cause Description
VM powered off Virtual machine intentionally or accidentally stopped.
Boot disk detached No operating system available to boot.
Incorrect boot order VM attempting to boot from the wrong device.
Storage unavailable Underlying storage pool inaccessible.
Guest operating system corruption Windows boot files damaged.
Recent configuration changes Hardware configuration modified before failure.

Recovery Procedure

Recovery Option 1 — Start the Virtual Machine

qm start 101

If the server boots successfully, continue with validation procedures.

Recovery Option 2 — Reattach Boot Disk

If the disk was intentionally detached for testing:

  • Open VM Hardware.
  • Reconnect the original virtual disk.
  • Verify the boot order.
  • Restart the virtual machine.

Recovery Option 3 — Restore Previous Snapshot

qm rollback 101 baseline-state

qm start 101

This restores the virtual machine to the baseline recovery point created during Playbook 0.

Recovery Option 4 — Restore from Tier 2 Backup

If the virtual machine cannot be recovered through snapshot restoration, proceed to PB1-S10 and restore the VM from the Tier 2 VZDump backup.

Validation Procedure

Verify Virtual Machine

qm status 101

Expected output:

status: running

Verify Windows Network Configuration

ipconfig /all

Verify Active Directory

dcdiag

Verify DNS

nslookup jcag-lab.local

Verify DHCP

Renew a client lease and confirm successful IP assignment from the Windows Server DHCP service. 

ipconfig /release

ipconfig /renew

Prevention & Best Practices

  • Always complete Playbook 0 before making infrastructure changes.
  • Create a snapshot before Windows updates, driver installations, or major configuration changes.
  • Maintain current Tier 2 VZDump backups for all production virtual machines.
  • Avoid making multiple configuration changes simultaneously.
  • Monitor storage utilization to prevent unexpected VM failures.
  • Verify VM health after every maintenance activity.
  • Document all infrastructure modifications through change records.
  • Review Proxmox task logs after maintenance to identify failed operations.

PB1-S02 — Active Directory Services Unavailable

Scenario Overview

Active Directory Domain Services (AD DS) provides centralized authentication, authorization, directory services, and Group Policy management throughout the JCAG-LAB environment. If AD DS becomes unavailable, domain users, computers, and integrated services may no longer authenticate, resulting in widespread service disruption.

This scenario demonstrates how to determine whether the issue originates from the Active Directory service itself, DNS, networking, or the underlying virtual machine.

This scenario is classified as Safe Lab Validation. The failure can be reproduced safely by stopping the Active Directory Domain Services service rather than damaging the operating system.

Business Impact

Affected Function Operational Impact
User Authentication Domain logins fail
Computer Authentication Workstations cannot authenticate
Group Policy Policies are not applied
Domain Services Directory queries fail
Administrative Tasks Domain management unavailable

Observable Symptoms

  • Users receive authentication errors.
  • "The domain is unavailable" messages appear.
  • Group Policy processing fails.
  • Domain computers cannot join the domain.
  • Applications depending on LDAP fail.
  • Administrative tools cannot connect to the domain controller.

Safe Lab Validation Procedure

Objective

Simulate an Active Directory outage without damaging the operating system or directory database.

Method — Stop Active Directory Domain Services

Open an elevated PowerShell session on the domain controller. 

Stop-Service NTDS

Attempt to authenticate from a domain workstation.

Expected observations:

  • Authentication fails.
  • Domain resources become unavailable.
  • LDAP queries fail.
Do not modify or delete the Active Directory database (NTDS.DIT). This playbook focuses on service-level recovery only.

Scenario 4 — DNS Service Failure

Domain Name System (DNS) is a critical dependency for Active Directory, authentication, name resolution, Group Policy processing, and numerous enterprise applications. Failure of the DNS service can affect nearly every system within the environment.

Business Impact

  • Users cannot locate domain resources.
  • Active Directory logons may fail.
  • Servers cannot resolve hostnames.
  • Applications experience connection failures.
  • Group Policy processing becomes unreliable.

Typical Symptoms

  • Ping by hostname fails.
  • Ping by IP address succeeds.
  • Domain controller cannot be located.
  • Windows login delays.
  • Applications report DNS lookup failures.

Safe Lab Validation

This failure can be safely reproduced within the isolated lab. 

Stop-Service DNS

Scenario 5 — DHCP Service Failure

When the DHCP service becomes unavailable, newly connected devices are unable to obtain valid IP addressing information. Existing clients continue operating until their leases expire.

Business Impact

  • Clients receive APIPA addresses.
  • New systems cannot join the network.
  • Voice devices may fail to register.
  • Operational downtime increases.

Typical Symptoms

  • 169.254.x.x addresses.
  • Unable to contact default gateway.
  • No DNS servers assigned.
  • Network connectivity unavailable.

Safe Lab Validation

ipconfig /all

Recovery Procedure

Start-Service DHCPServer

Verify:

  • Address pool available.
  • Clients successfully renew leases.

Scenario 7 — Proxmox Host Unavailable

The Proxmox Virtual Environment host provides the virtualization platform for all enterprise infrastructure within the JCAG-LAB. If the host becomes unavailable, every virtual machine hosted on the platform is affected, including Active Directory, Wazuh SIEM, Peppermint, Cisco Modeling Labs, and supporting services.

Business Impact

  • All hosted virtual machines become unavailable.
  • Authentication services are interrupted.
  • Monitoring and alerting stop.
  • Automation workflows fail.
  • Administrative access to hosted services is lost.

Typical Symptoms

  • Unable to access the Proxmox Web UI.
  • SSH connection to the host fails.
  • Virtual machines become unreachable.
  • Monitoring alerts indicate multiple systems offline.
  • Hosted services become unavailable simultaneously.

Safe Lab Validation

Do NOT intentionally power off the Proxmox host during normal lab operation. Instead, safely simulate management access failure by stopping the web proxy service. 

systemctl stop pveproxy

This affects only the web management interface while virtual machines continue running.

Restore management access:

systemctl start pveproxy

  1. Verify physical power.

  3. Restart failed management services.

  5. Verify storage availability.
  6. Confirm all virtual machines are operational.

Scenario 8 — Storage Pool Capacity Exhaustion

Storage exhaustion is one of the most common causes of virtual infrastructure instability. This scenario validates storage monitoring across the three-tier storage architecture deployed within JCAG-LAB.

Applicable Storage Design

Tier Storage Purpose
Tier 0 local-lvm (SSD) Performance-critical virtual machines
Tier 1 local-lvm-hdd-01 General-purpose VM storage
Tier 2 HDD-02 Directory Storage VZDump backup repository

Business Impact

  • VM creation fails.
  • Snapshots fail.
  • Backups fail.
  • Disk expansion unavailable.
  • Virtual machines may pause unexpectedly.

Safe Lab Validation

Do not intentionally fill storage to 100%. Instead, create a temporary file that can be safely removed afterward. 

fallocate -l 5G /tmp/storage-test.img

Remove after testing:

rm /tmp/storage-test.img

Diagnostic Procedure

 lvs

 vgs

 pvs

 pvesm status

Example output:

Name                 Type     Status

 local-lvm            lvmthin  active

Recovery Procedure

  • Remove unnecessary snapshots.
  • Delete temporary files.
  • Move non-critical VMs to Tier 1 storage.
  • Ensure backups remain on Tier 2.
  • Expand storage if additional capacity is available.

Server & Storage Recovery Decision Matrix

The following matrix provides a structured approach for selecting the appropriate recovery action based on the observed failure.

Failure Type Primary Recovery Method Escalation Path
Windows boot failure Startup Repair / Snapshot Restore from Backup
Active Directory failure Restart AD DS Authoritative Restore
DNS failure Restart DNS Service Restore Configuration
DHCP failure Restart DHCP Service Restore DHCP Database
SMB failure Restart LanmanServer Restore Server Snapshot
Proxmox management failure Restart Management Services Host Recovery
Storage capacity issue Free Capacity Expand Storage
Virtual disk corruption Snapshot Rollback Restore Backup
Backup recovery Restore VZDump Archive Disaster Recovery
Web application outage Restart Container Restore VM

Operational Troubleshooting Workflow

Step 1 — Assess Business Impact

Determine which infrastructure services are affected and identify the scope of the outage.

  • Single virtual machine
  • Multiple services
  • Entire virtualization host
  • Authentication outage
  • Storage failure

Step 2 — Collect Diagnostic Information

hostnamectl

 uptime

 df -h

 free -h

 journalctl -xe

 systemctl --failed

 qm list

Step 3 — Determine Root Cause

Identify whether the incident originated from:

  • Operating system
  • Application service
  • Storage subsystem
  • Virtualization platform
  • Network connectivity
  • Configuration error

Step 4 — Recover Service

Implement the least disruptive recovery method.

  1. Restart service
  2. Restart application
  3. Restart VM
  4. Rollback snapshot
  5. Restore backup

Step 5 — Validate Recovery

  • Users can authenticate.
  • Applications respond normally.
  • Monitoring returns to normal.
  • No critical events remain.
  • Business service restored.

Key Takeaways

  • Server failures should be approached using a structured troubleshooting methodology rather than assumptions.
  • Enterprise services such as Active Directory, DNS, DHCP, and SMB have dependencies that should always be validated before implementing recovery actions.
  • Snapshots provide rapid rollback capability but are not a replacement for VZDump backups stored on the dedicated Tier 2 backup storage.
  • The three-tier storage architecture (Tier 0 SSD, Tier 1 HDD VM Storage, Tier 2 Backup Storage) supports both operational performance and disaster recovery objectives.
  • Safe failure simulations within the JCAG-LAB provide practical experience while protecting the integrity of the production lab environment.
  • Recovery procedures should always conclude with service validation and documentation to ensure the environment has returned to a known-good operational state.

Playbook Completion Outcome

Upon completion of this playbook: Capable of diagnosing, recovering, and validating failures affecting enterprise servers, virtualization hosts, storage resources, Active Directory services, DNS, DHCP, SMB file services, Docker-hosted applications, and virtual machine backups within the JCAG-LAB enterprise environment. The procedures documented throughout this playbook establish a repeatable operational methodology aligned with common break/fix practices used by infrastructure support teams, systems administrators, and data center operations personnel.

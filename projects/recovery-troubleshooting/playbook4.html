Infrastructure Recovery & Troubleshooting

Playbook 4: Operating System Deployment & Recovery

Overview

Operating systems form the foundation of every enterprise service deployed within the JCAG-LAB environment. Network infrastructure may be fully operational, but business services remain unavailable if critical operating systems fail to boot, lose configuration integrity, or experience service failures.

This playbook documents the operational procedures used to recover Windows and Linux operating systems following planned maintenance, configuration errors, software failures, storage issues, or virtual machine corruption. The objective is to restore service availability while preserving data integrity and following structured enterprise recovery procedures.

Unlike deployment documentation, this playbook focuses on incident response and service restoration. Each recovery procedure follows a repeatable methodology designed to minimize downtime, validate system integrity, and safely return systems to production within the JCAG-LAB enterprise environment.

Learning Objectives

  • Identify operating system failures affecting enterprise infrastructure.
  • Perform structured recovery of Windows and Linux virtual machines.
  • Restore enterprise services while minimizing operational downtime.
  • Validate operating system health following recovery activities.
  • Document recovery procedures using repeatable enterprise operational practices.

Windows Server 2019 Recovery Philosophy

Windows Server 2019 is the most critical operating system within the JCAG-LAB environment. It hosts Active Directory Domain Services (AD DS), DNS, DHCP, and SMB file services that support authentication, name resolution, IP address allocation, and shared resources throughout the enterprise network.

Recovery activities should prioritize preserving Active Directory integrity and restoring infrastructure services before addressing secondary application issues. Whenever possible, recovery should begin with service validation before considering snapshot rollback or full virtual machine restoration.

Recovery Priority

  1. Verify virtual machine health.
  2. Confirm operating system boot status.
  3. Validate Active Directory functionality.
  4. Verify DNS services.
  5. Verify DHCP services.
  6. Verify SMB file sharing.
  7. Validate client authentication.

Primary Validation Commands

Windows 11 Client Recovery Philosophy

Windows 11 systems represent enterprise endpoints used to access infrastructure services hosted within the JCAG-LAB environment. Recovery procedures focus on restoring workstation functionality while confirming successful communication with domain services and network resources.

Because workstation failures often originate from infrastructure dependencies, client recovery should always begin by validating network connectivity, authentication, DNS resolution, and Group Policy processing before modifying the local operating system.

Recovery Priorities

  • Verify IP addressing.
  • Verify DNS resolution.
  • Confirm Active Directory communication.
  • Validate user authentication.
  • Verify mapped network resources.

Scenario 1 – Windows Server 2019 Fails to Boot

Windows Server 2019 hosts the core enterprise services within the JCAG-LAB environment, including Active Directory Domain Services (AD DS), DNS, DHCP, and SMB file sharing. If the operating system fails to boot, all dependent enterprise services become unavailable, resulting in widespread authentication, name resolution, and network service failures.

Business Impact

  • Active Directory authentication unavailable.
  • DNS and DHCP services offline.
  • SMB file shares inaccessible.
  • Enterprise users unable to authenticate.
  • Monitoring and automation platforms may lose connectivity to domain resources.

Symptoms

  • Virtual machine displays Windows Recovery Environment.
  • Continuous reboot cycle.
  • Blue Screen of Death (BSOD).
  • Operating system freezes during startup.
  • VM powers on but never reaches the logon screen.

Initial Assessment

Before attempting operating system recovery, verify that the problem is not caused by the virtualization platform or storage subsystem. 


qm status 101

qm config 101

Example output:


status: running
memory: 4096
cores: 2
boot: order=scsi0

Diagnostic Procedure

Review the virtual machine console within Proxmox.

Verify storage availability. 


lsblk

pvesm status

Review Windows recovery options from the VM console.

  • Startup Repair
  • Safe Mode
  • Command Prompt
  • System Restore

Root Cause Analysis

  • Operating system corruption.
  • Boot Configuration Data (BCD) corruption.
  • Storage interruption.
  • Failed Windows Update.
  • Filesystem corruption.
  • Unexpected shutdown.

Recovery Procedure

Attempt Startup Repair from the Windows Recovery Environment.

If Startup Repair is unsuccessful, repair the boot configuration. 


bootrec /fixmbr

bootrec /fixboot

bootrec /scanos

bootrec /rebuildbcd

If recovery remains unsuccessful, restore the most recent snapshot created during Playbook 0. 


qm rollback 101 pre-maintenance

If no valid snapshot exists, restore the latest VZDump backup stored on Tier 2 backup storage.

Validation

  • Windows boots successfully.
  • Administrator logon available.
  • Network connectivity restored.
  • No critical errors recorded after startup.

Scenario 2 – Active Directory Services Fail to Start

Following a successful operating system boot, Active Directory Domain Services may fail to initialize due to service corruption, dependency failures, database issues, or DNS problems. Since Active Directory is the identity provider for the JCAG-LAB environment, failure prevents user authentication and domain-based resource access.

Business Impact

  • Users unable to log into the domain.
  • Group Policy processing fails.
  • DNS integration unavailable.
  • Authentication requests rejected.
  • Enterprise services relying on Kerberos become unavailable.

Symptoms

  • Domain logon unavailable.
  • Authentication errors.
  • Netlogon service failures.
  • Domain controller unavailable.
  • Directory Services errors in Event Viewer.

Diagnostic Procedure

Verify Active Directory services. 


Get-Service NTDS

Get-Service Netlogon

Get-Service DNS

Run Domain Controller diagnostics. 


dcdiag

Example output:


Starting test: Connectivity

Passed Test Connectivity

Starting test: Advertising

Passed Test Advertising

Review Directory Services event logs. 


Get-EventLog -LogName "Directory Service" -Newest 20

Root Cause Analysis

  • NTDS service stopped.
  • Database corruption.
  • DNS dependency failure.
  • SYSVOL unavailable.
  • Unexpected shutdown.

Recovery Procedure

Restart Active Directory services. 


Restart-Service NTDS

Restart-Service Netlogon

Restart-Service DNS

Verify Active Directory health. 


dcdiag

repadmin /replsummary

Confirm client authentication using a domain workstation. 


whoami

gpupdate /force

Validation

  • Domain logon successful.
  • DNS operational.
  • Kerberos authentication restored.
  • Group Policy applied successfully.

Scenario 3 – Windows Update Causes Service Failure

Windows Updates improve security and reliability but occasionally introduce unexpected service interruptions or configuration changes. Within the JCAG-LAB environment, failed updates may affect Active Directory, DNS, DHCP, file sharing, or application compatibility.

Business Impact

  • Critical infrastructure services unavailable.
  • Unexpected server reboot.
  • Application incompatibility.
  • Authentication delays.
  • Operational downtime.

Symptoms

  • Services fail immediately after reboot.
  • Windows Update reports installation failure.
  • Missing or disabled services.
  • Performance degradation.
  • Event Viewer records update-related errors.

Diagnostic Procedure

Review installed updates. 


Get-HotFix

Review Windows Update history. 


Get-WindowsUpdateLog

Review recent System events. 


Get-EventLog -LogName System -Newest 50

Verify infrastructure services. 


Get-Service DNS

Get-Service DHCPServer

Get-Service NTDS

Root Cause Analysis

  • Incomplete Windows Update installation.
  • Driver incompatibility.
  • Service dependency modification.
  • Unexpected reboot during update.
  • Corrupted system files.

Recovery Procedure

Restart affected services. 


Restart-Service DNS

Restart-Service DHCPServer

Restart-Service NTDS

If the issue persists, uninstall the most recent update. 


wusa /uninstall /kb:<KB_Number>

If operating system stability cannot be restored, roll back to the approved pre-maintenance snapshot created before patching. 


qm rollback 101 pre-maintenance

If snapshot recovery is unavailable or unsuitable, restore the latest verified Tier 2 VZDump backup following the procedures established in Playbook 0.

Validation

  • Windows boots normally.
  • All enterprise services operational.
  • No critical update-related errors remain.
  • Domain clients authenticate successfully.
  • System ready for production operation.

Scenario 4 – Domain Trust Relationship Failure

A secure trust relationship exists between every domain-joined workstation and the Active Directory domain. If this relationship becomes invalid, users may be unable to authenticate using domain credentials even though the workstation, network connectivity, and domain controller remain operational.

Within the JCAG-LAB environment, this scenario primarily affects Windows 11 enterprise endpoints located in VLAN 10 while the Windows Server 2019 Domain Controller continues providing Active Directory, DNS, and DHCP services.

Business Impact

  • Domain users cannot log on.
  • Group Policy processing fails.
  • Mapped network drives become unavailable.
  • Kerberos authentication fails.
  • User productivity is interrupted.

Symptoms

  • "The trust relationship between this workstation and the primary domain failed."
  • Domain logon rejected.
  • Computer account authentication errors.
  • Group Policy failures.
  • Access to shared resources denied.

Diagnostic Procedure

Verify network connectivity. 


ipconfig /all

ping 10.10.100.11

nslookup jcag-lab.local

Verify the secure channel. 


Test-ComputerSecureChannel

Example output:


False

Verify computer account information. 


nltest /sc_verify:jcag-lab.local

Root Cause Analysis

  • Computer account password mismatch.
  • Snapshot restored without resetting trust.
  • Active Directory computer object corruption.
  • Duplicate computer account.
  • Manual domain configuration changes.

Recovery Procedure

Attempt secure channel repair. 


Test-ComputerSecureChannel -Repair -Credential jcag-lab\Administrator

If repair is unsuccessful, remove the workstation from the domain, reboot, and rejoin the domain.

Verify successful authentication after rejoining. 


whoami

gpupdate /force

Validation

  • Domain authentication successful.
  • Secure channel restored.
  • Group Policy updated successfully.
  • Network resources accessible.

Scenario 5 – Windows Service Startup Failure (DNS, DHCP & File Services)

Enterprise services hosted by Windows Server 2019 depend on individual Windows services starting successfully during system initialization. Failure of DNS, DHCP, or SMB services may occur following configuration changes, software updates, dependency failures, or unexpected shutdowns.

The objective is to restore the affected services while minimizing disruption to the remaining infrastructure.

Business Impact

  • Name resolution unavailable.
  • DHCP clients unable to obtain IP addresses.
  • File shares inaccessible.
  • Authentication delays.
  • Enterprise applications may fail to communicate.

Symptoms

  • DNS queries fail.
  • DHCP lease requests time out.
  • Shared folders unavailable.
  • Service status shows "Stopped."
  • Service startup failures recorded in Event Viewer.

Diagnostic Procedure

Verify service status. 


Get-Service DNS

Get-Service DHCPServer

Get-Service LanmanServer

Review service dependencies. 


sc qc DNS

sc qc DHCPServer

Review recent system events. 


Get-EventLog -LogName System -Newest 30

Example output:


Service Control Manager

The DNS Server service terminated unexpectedly.

Root Cause Analysis

  • Service dependency failure.
  • Configuration corruption.
  • Port conflicts.
  • Windows Update side effects.
  • Unexpected shutdown.

Recovery Procedure

Restart affected services. 


Restart-Service DNS

Restart-Service DHCPServer

Restart-Service LanmanServer

If services continue to fail, review Event Viewer and restore configuration from the most recent known-good backup or snapshot following Playbook 0.

Validation

  • DNS resolves internal hosts.
  • DHCP leases issued successfully.
  • SMB shares accessible.
  • Services remain running after reboot.

Scenario 6 – Windows Endpoint Profile & Login Failure

Windows user profiles occasionally become corrupted following unexpected shutdowns, failed updates, storage issues, or synchronization problems. Users may be unable to log in or may receive temporary profiles that prevent access to their normal desktop environment.

Within JCAG-LAB, this scenario applies to Windows 11 domain workstations used for enterprise administration and security monitoring activities.

Business Impact

  • User unable to access workstation.
  • Temporary profile loaded.
  • User settings unavailable.
  • Administrative tasks interrupted.
  • Reduced operational productivity.

Symptoms

  • "User Profile Service failed the sign-in."
  • Temporary desktop loaded.
  • Missing desktop files.
  • Group Policy processing errors.
  • Application settings reset.

Diagnostic Procedure

Verify Event Viewer. 


eventvwr.msc

Verify user profile directories. 


C:\Users\

Verify Group Policy processing. 


gpresult /r

gpupdate /force

Verify domain communication. 


whoami

hostname

ping 10.10.100.11

Root Cause Analysis

  • Corrupted user profile.
  • Failed Group Policy processing.
  • Insufficient disk space.
  • Domain authentication issues.
  • Windows Update interruption.

Recovery Procedure

Attempt profile repair by verifying profile integrity and correcting registry profile mappings if necessary.

If the profile remains unusable, create a new user profile and migrate user data after confirming successful domain authentication.

Reboot the workstation and verify normal logon behavior.

Validation

  • User successfully logs into the domain.
  • Correct desktop profile loaded.
  • Group Policy applied successfully.
  • User files accessible.
  • Applications function normally.

Scenario 7 – Ubuntu Server Boot Failure (Wazuh / n8n / Docker Host)

Several critical infrastructure services within the JCAG-LAB environment are hosted on Ubuntu Server virtual machines, including the Wazuh Manager, n8n automation platform, and Docker-based applications. A boot failure affecting these systems can interrupt security monitoring, workflow automation, and containerized services.

Recovery should begin by determining whether the failure originates from the virtualization layer, operating system, storage subsystem, or application services before considering snapshot rollback or full virtual machine restoration.

Business Impact

  • Wazuh Manager unavailable.
  • n8n automation workflows stop executing.
  • Docker containers unavailable.
  • Security event collection interrupted.
  • Incident response automation disabled.

Symptoms

  • Ubuntu fails to reach the login prompt.
  • Kernel panic displayed during boot.
  • Emergency Mode prompt appears.
  • Repeated boot loop.
  • VM remains unresponsive after startup.

Initial Assessment

Verify the virtual machine is running and that the underlying storage is healthy. 


qm status 105

qm config 105

pvesm status

Example output:


status: running
boot: order=scsi0
memory: 4096

Diagnostic Procedure

Review the VM console from the Proxmox Web UI and determine whether the system reaches the GRUB menu or Emergency Mode.

If shell access is available, collect system logs. 


journalctl -xb

systemctl --failed

dmesg | less

Root Cause Analysis

  • Filesystem corruption.
  • Kernel update failure.
  • Incorrect GRUB configuration.
  • Storage interruption.
  • Unexpected power loss.

Recovery Procedure

Repair the operating system where possible before restoring from snapshot. 


fsck -f /dev/sda2

update-grub

reboot

If recovery is unsuccessful, restore the virtual machine using the approved snapshot or Tier 2 VZDump backup created during Playbook 0.

Validation

  • Ubuntu boots normally.
  • SSH access restored.
  • Network connectivity verified.
  • Application services operational.

Scenario 8 – Linux Filesystem & Disk Space Recovery

Disk utilization problems are a common cause of Linux service failures. Container logs, package caches, application data, and security event logs may consume available storage, preventing services from starting correctly.

Within JCAG-LAB this scenario is particularly applicable to the Wazuh Manager, which continuously ingests security telemetry, and Docker hosts that maintain container images and log files.

Business Impact

  • Applications fail to start.
  • Docker containers stop unexpectedly.
  • Wazuh indexing interrupted.
  • Log collection halted.
  • Automation services unavailable.

Symptoms

  • "No space left on device."
  • Filesystem mounted read-only.
  • High disk utilization.
  • System performance degradation.
  • Repeated service failures.

Initial Assessment


df -h

lsblk

mount

Example output:


Filesystem      Size Used Avail Use%
/dev/sda2        80G 79G  1G   99%

Diagnostic Procedure

Identify directories consuming excessive storage. 


du -sh /*

du -sh /var/*

journalctl --disk-usage

Review Docker storage. 


docker system df

Root Cause Analysis

  • Oversized log files.
  • Unreleased Docker images.
  • Large security event databases.
  • Package cache accumulation.
  • Filesystem corruption.

Recovery Procedure


journalctl --vacuum-time=7d

apt clean

docker system prune -a

reboot

If filesystem corruption is detected, boot into maintenance mode and perform a filesystem consistency check before returning the server to production.

Validation

  • Disk utilization reduced.
  • Filesystem mounted read-write.
  • Application services restored.
  • No storage-related alerts remain.

Scenario 9 – Virtual Machine Recovery Using Proxmox Snapshots & Backups

When operating system recovery is unsuccessful, the virtual machine should be restored using the recovery mechanisms established in Playbook 0. Recovery should always follow the least disruptive method, beginning with snapshots before proceeding to full backup restoration.

Business Impact

  • Enterprise services unavailable.
  • Extended maintenance window.
  • Potential configuration loss.
  • Temporary service interruption.

Recovery Strategy

Recovery Method Use Case
Snapshot Rollback Recent configuration changes or failed maintenance.
VZDump Restore VM corruption or complete operating system failure.

Snapshot Recovery


qm listsnapshot 105

qm rollback 105 pre-maintenance

qm start 105

Backup Recovery


qmrestore \
/mnt/pve/HDD-02/dump/vzdump-qemu-105.vma.zst \
105

Backups must be restored from the dedicated Tier 2 backup storage repository, never from Tier 0 or Tier 1 production storage.

Validation

  • VM starts successfully.
  • Correct snapshot restored.
  • Applications operational.
  • No configuration inconsistencies detected.

Key Takeaways

  • Operating system failures should always be approached using a structured diagnostic methodology before initiating recovery.
  • Windows Server 2019 hosts multiple critical enterprise services; service-level recovery should be attempted before restoring the entire virtual machine.
  • Ubuntu infrastructure servers supporting Wazuh, n8n, and Docker require verification of operating system health before application troubleshooting begins.
  • Snapshots provide rapid rollback capability, while Tier 2 VZDump backups remain the authoritative recovery mechanism following severe virtual machine corruption or data loss.
  • Every recovery operation must conclude with comprehensive validation of operating system functionality, enterprise services, network connectivity, monitoring, and documentation.

Playbook Completion Outcome

Upon completion of this playbook: Capable of diagnosing and recovering Windows Server 2019, Windows 11, and Ubuntu Server operating system failures within the JCAG-LAB environment. This includes repairing boot issues, restoring Active Directory services, recovering Windows service failures, resolving Linux filesystem problems, performing Proxmox snapshot and Tier 2 VZDump backup restorations, validating enterprise application functionality, and returning infrastructure systems to operational status using a structured enterprise recovery methodology.

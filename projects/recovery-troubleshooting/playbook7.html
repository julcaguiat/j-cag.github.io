Playbook 7: Incident Management

Overview

Enterprise security operations extend beyond detecting malicious activity. They require a structured process for collecting telemetry, identifying abnormal behavior, validating security events, containing affected systems, restoring services, documenting incidents, and implementing improvements to reduce future risk.

Within the JCAG-LAB environment, security operations are performed using an integrated monitoring and automation platform consisting of Windows endpoints, Sysmon telemetry, Wazuh SIEM, n8n workflow automation, Peppermint IT Service Management, Windows Server 2019 infrastructure services, and Proxmox Virtual Environment. Together, these components simulate the operational workflow found within enterprise Security Operations Centers (SOC) and Infrastructure Operations teams.

This playbook documents the operational procedures used to investigate, validate, contain, recover, and document security incidents affecting the JCAG-LAB environment. Each scenario emphasizes repeatable operational methodology rather than isolated tool usage, reinforcing the importance of consistent incident response procedures and cross-platform troubleshooting.

Learning Objectives

  • Understand the security operations architecture deployed within the JCAG-LAB environment.
  • Follow a structured incident response process from detection through recovery.
  • Investigate security events using Wazuh, Sysmon, Windows Event Logs, and supporting infrastructure.
  • Correlate security telemetry with enterprise services to determine operational impact.
  • Document security incidents using repeatable operational procedures consistent with enterprise SOC practices.

Detection Engineering Philosophy

Detection engineering is the process of developing, validating, and maintaining security detections that identify malicious activity while minimizing false positives. Within the JCAG-LAB environment, detection engineering focuses on building reliable operational visibility using Windows Event Logs, Sysmon telemetry, and Wazuh correlation rules.

Rather than relying solely on default signatures, detections should be continuously reviewed, validated, and refined based on observed system behavior. Each detection should produce actionable information that supports rapid investigation and informed decision-making.

Effective detection engineering within the JCAG-LAB environment follows several core principles:

  • Collect high-quality endpoint telemetry.
  • Normalize and centralize security events.
  • Reduce unnecessary alert noise.
  • Prioritize actionable detections.
  • Validate every detection through controlled laboratory testing whenever practical.
  • Document detection logic and expected response procedures.

Safe failure simulations performed throughout this playbook provide opportunities to validate existing detections, tune alert thresholds, and improve incident response workflows without affecting production systems.

Scenario 1 – Multiple Failed Logon Attempts (Windows Event ID 4625)

Windows Security Event ID 4625 indicates that an account failed to authenticate. Although individual failures are common during normal operations, repeated or high-volume authentication failures may indicate password guessing, brute-force activity, misconfigured services, or user credential issues.

Within the JCAG-LAB environment, Windows Event Logs generated by the Windows 11 endpoint and Windows Server 2019 domain controller are collected by the Wazuh Agent, correlated by the Wazuh Manager, and may trigger automated workflows through n8n and Peppermint ITSM.

Business Impact

  • Potential brute-force attack against enterprise accounts.
  • Increased authentication failures.
  • Possible account lockouts.
  • Unauthorized access attempts.
  • Security monitoring alerts generated.

Symptoms

  • Repeated Wazuh authentication alerts.
  • Windows Security Event ID 4625 entries.
  • Multiple failed login attempts.
  • User reports inability to authenticate.
  • Peppermint incident automatically created.

Initial Assessment

Confirm that Wazuh generated the authentication alert and identify the affected endpoint or domain controller.

Review Windows Security Event Logs. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4625
} -MaxEvents 20

Example output:


Event ID : 4625

An account failed to log on.

Diagnostic Procedure

Identify the affected account, source workstation, logon type, and originating IP address.

Verify that Wazuh received the event and generated the corresponding alert.

Review recent authentication activity on Windows Server 2019. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4625
}

Determine whether the failures originated from a legitimate user, scheduled task, mapped drive, service account, or external source.

Root Cause Analysis

  • User entered an incorrect password.
  • Password recently changed.
  • Scheduled task using outdated credentials.
  • Mapped drive using cached credentials.
  • Brute-force attack.
  • Unauthorized authentication attempt.

Recovery Procedure

  • Confirm user identity.
  • Reset credentials if necessary.
  • Correct scheduled task or service account credentials.
  • Investigate repeated authentication failures.
  • Document the incident.

Validation

  • No additional Event ID 4625 entries observed.
  • User successfully authenticates.
  • Wazuh alert resolved.
  • Infrastructure services operational.
  • Incident documentation completed.

Scenario 2 – Privileged Account Lockout

Administrative account lockouts have a greater operational impact than standard user accounts because they may prevent administrators from managing Active Directory, virtualization infrastructure, network services, or monitoring platforms. Rapid investigation is required to determine whether the lockout was caused by normal administrative activity or malicious authentication attempts.

Business Impact

  • Administrative management interrupted.
  • Critical infrastructure unavailable for maintenance.
  • Potential security compromise.
  • Delayed incident response.
  • Operational risk increased.

Symptoms

  • Administrator account locked.
  • Repeated failed authentication attempts.
  • Domain administrator unable to log in.
  • Authentication alerts generated.
  • Help desk reports administrative access issues.

Initial Assessment

Identify the affected account and determine when the lockout occurred.

Review account lockout events. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4740
}

Example output:


A user account was locked out.

Diagnostic Procedure

Review authentication failures immediately preceding the lockout. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4625
}

Determine whether the account is being used by:

  • Scheduled tasks.
  • Windows services.
  • Mapped network drives.
  • Remote Desktop sessions.
  • Administrative scripts.

Review the Wazuh alert timeline to identify the originating system.

Root Cause Analysis

  • Incorrect administrator password.
  • Cached credentials.
  • Expired password.
  • Scheduled task using old credentials.
  • Malicious authentication attempts.

Recovery Procedure

  • Unlock the affected account.
  • Reset credentials if required.
  • Update scheduled tasks and services.
  • Validate administrative access.
  • Continue monitoring for additional failures.

Validation

  • Administrative login successful.
  • No additional lockouts observed.
  • Infrastructure administration restored.
  • Authentication alerts resolved.
  • Incident documentation completed.

Scenario 3 – Suspicious PowerShell Execution

PowerShell is an essential administrative tool but is also frequently abused by attackers because of its ability to execute scripts, access system resources, and interact with Windows management interfaces. Suspicious PowerShell activity should always be investigated to determine whether execution was authorized, expected, or potentially malicious.

Within the JCAG-LAB environment, Sysmon process creation events and Windows PowerShell logs are collected by the Wazuh Agent and analyzed by the Wazuh Manager for detection engineering and alert generation.

Business Impact

  • Potential malicious code execution.
  • Credential theft attempts.
  • Unauthorized system modification.
  • Persistence mechanisms installed.
  • Security incident initiated.

Symptoms

  • Wazuh PowerShell alert generated.
  • Sysmon Process Creation events recorded.
  • Unexpected PowerShell windows.
  • Encoded PowerShell commands detected.
  • Automation workflow initiated.

Initial Assessment

Review the PowerShell command line captured by Sysmon.

Review recent Sysmon process creation events. 


Get-WinEvent `
-LogName "Microsoft-Windows-Sysmon/Operational" `
-MaxEvents 20

Review Windows PowerShell Operational logs. 


Get-WinEvent `
-LogName "Microsoft-Windows-PowerShell/Operational" `
-MaxEvents 20

Diagnostic Procedure

Identify:

  • Executing user account.
  • Parent process.
  • Complete command line.
  • Execution time.
  • Affected endpoint.

Correlate Sysmon telemetry with the corresponding Wazuh alert and determine whether the execution originated from legitimate administrative activity or an unexpected source.

Root Cause Analysis

  • Authorized administrative script.
  • Software installation.
  • Scheduled automation.
  • Encoded PowerShell execution.
  • Potential malware activity.

Recovery Procedure

  • Terminate unauthorized PowerShell sessions if required.
  • Collect forensic evidence before making changes.
  • Isolate the affected endpoint if compromise is suspected.
  • Review persistence mechanisms.
  • Document investigative findings.

Validation

  • Unauthorized PowerShell execution stopped.
  • Endpoint operating normally.
  • No additional suspicious activity detected.
  • Monitoring platform continues collecting telemetry.
  • Incident documentation completed.

Scenario 4 – Unauthorized Local Administrator Creation

The unauthorized creation of a local administrator account is a high-severity security event because it may provide persistent privileged access to an endpoint outside of Active Directory control. Whether the account was created through malicious activity, misconfiguration, or legitimate administration, the event must be investigated immediately.

Within the JCAG-LAB environment, account creation events generated by Windows 11 endpoints are collected by the Wazuh Agent, correlated by the Wazuh Manager, and can trigger automated incident workflows through n8n and Peppermint ITSM.

Business Impact

  • Unauthorized privileged access established.
  • Administrative accountability reduced.
  • Potential persistence mechanism created.
  • Endpoint security compromised.
  • Incident response required.

Symptoms

  • Unexpected administrator account appears.
  • Wazuh privilege escalation alert generated.
  • Windows Security events indicate account creation.
  • New account added to the Administrators group.
  • Peppermint incident automatically generated.

Initial Assessment

Review recent account management events from the Windows Security log. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4720
} -MaxEvents 20

Review membership changes to the local Administrators group. 


Get-WinEvent `
-FilterHashtable @{
LogName='Security'
Id=4732
} -MaxEvents 20

Example output:


A user account was created.

A member was added to a security-enabled local group.

Diagnostic Procedure

Identify:

  • User account that created the administrator account.
  • Time of creation.
  • Endpoint involved.
  • Associated logon session.
  • Related Wazuh alerts.

Review current local administrator membership. 


net localgroup administrators

Root Cause Analysis

  • Authorized administrative maintenance.
  • Software installation requiring elevation.
  • Configuration error.
  • Unauthorized privilege escalation.
  • Compromised administrator credentials.

Recovery Procedure

  • Verify whether account creation was authorized.
  • Disable or remove unauthorized accounts.
  • Reset privileged credentials if compromise is suspected.
  • Review additional administrative changes.
  • Document investigative findings.

Validation

  • Unauthorized account removed or disabled.
  • Local Administrators group verified.
  • No additional privilege escalation detected.
  • Endpoint security restored.
  • Incident documentation completed.

Scenario 5 – Windows Firewall Disabled

Disabling Windows Defender Firewall significantly reduces endpoint security by allowing unrestricted network communication. Although administrators may temporarily disable the firewall during troubleshooting, unexpected firewall changes should always be treated as a potential security incident until validated.

Within the JCAG-LAB environment, configuration changes affecting Windows Defender Firewall should generate Windows Event Logs and corresponding Wazuh alerts for investigation.

Business Impact

  • Endpoint exposure increases.
  • Unauthorized inbound connections become possible.
  • Network segmentation effectiveness reduced.
  • Detection surface altered.
  • Security policy violation.

Symptoms

  • Firewall reported as disabled.
  • Wazuh configuration alert generated.
  • Unexpected inbound connectivity.
  • Security policy non-compliance.
  • Administrative change not documented.

Initial Assessment

Verify Windows Defender Firewall status. 


Get-NetFirewallProfile

Example output:


Domain  : False

Private : False

Public  : False

Review recent security configuration events. 


Get-WinEvent `
-LogName Security `
-MaxEvents 50

Diagnostic Procedure

Determine:

  • Who disabled the firewall.
  • When the change occurred.
  • Whether change control documentation exists.
  • Whether additional security settings were modified.
  • Whether suspicious network activity occurred afterward.

Review Wazuh alerts associated with the endpoint to identify correlated configuration changes.

Root Cause Analysis

  • Authorized maintenance.
  • Troubleshooting activity.
  • Administrative error.
  • Malware attempting to bypass defenses.
  • Unauthorized configuration modification.

Recovery Procedure

Re-enable Windows Defender Firewall using administrative PowerShell. 


Set-NetFirewallProfile `
-Profile Domain,Private,Public `
-Enabled True

Verify firewall status. 


Get-NetFirewallProfile

Continue monitoring for additional unauthorized configuration changes.

Validation

  • Firewall enabled on all profiles.
  • Security policy restored.
  • No additional unauthorized changes detected.
  • Endpoint remains reachable through approved services.
  • Incident documentation completed.

Scenario 6 – Sysmon Detects Suspicious Process Activity

Sysmon provides enhanced visibility into endpoint activity by recording detailed process creation events, parent-child process relationships, command-line arguments, network connections, and other telemetry not normally available in standard Windows logging. Suspicious process activity should be investigated to determine whether execution represents legitimate administrative activity or potential malicious behavior.

Business Impact

  • Potential malware execution.
  • Unauthorized software launched.
  • Privilege escalation attempts.
  • Lateral movement activity.
  • Security investigation required.

Symptoms

  • Sysmon Event ID 1 generated.
  • Unexpected executable launched.
  • Wazuh process creation alert.
  • High-risk parent-child relationship detected.
  • n8n automation workflow initiated.

Initial Assessment

Review recent Sysmon Process Creation events. 


Get-WinEvent `
-LogName "Microsoft-Windows-Sysmon/Operational" `
-MaxEvents 30

Review Wazuh alerts associated with the endpoint.

Diagnostic Procedure

Identify:

  • Executable name.
  • Full command line.
  • Parent process.
  • User account.
  • Process identifier (PID).
  • Execution timestamp.
  • Endpoint hostname.

Correlate Sysmon telemetry with:

  • Windows Security Events.
  • PowerShell Operational Logs.
  • Wazuh correlation rules.
  • Peppermint incident records.

Determine whether the observed activity corresponds to legitimate administrative work, scheduled automation, software installation, or potentially malicious execution.

Root Cause Analysis

  • Authorized application installation.
  • Administrative PowerShell execution.
  • Scheduled maintenance task.
  • Suspicious executable launched.
  • Potential malware or attacker activity.

Recovery Procedure

  • Preserve forensic evidence before making system changes.
  • Terminate unauthorized processes when appropriate.
  • Isolate the endpoint if compromise is suspected.
  • Review persistence mechanisms and startup locations.
  • Continue monitoring for recurring activity.

Validation

  • Suspicious process no longer executing.
  • No recurring Sysmon alerts observed.
  • Endpoint operating normally.
  • Monitoring pipeline functioning correctly.
  • Incident investigation completed and documented.

Scenario 7 – Malware Containment Procedure

Once malicious activity has been confirmed or is strongly suspected, the priority shifts from detection to containment. The primary objective is to prevent lateral movement, preserve forensic evidence, and minimize disruption to the remainder of the JCAG-LAB infrastructure while maintaining the integrity of the investigation.

Within the JCAG-LAB environment, containment focuses on the Windows 11 endpoint generating the alert while ensuring that Windows Server 2019, Wazuh Manager, n8n, Peppermint, and the Proxmox virtualization platform remain operational for continued monitoring and incident coordination.

Business Impact

  • Potential compromise of endpoint systems.
  • Risk of credential theft.
  • Possible lateral movement toward infrastructure services.
  • Loss of endpoint availability during containment.
  • Incident response procedures initiated.

Symptoms

  • High-severity Wazuh alert.
  • Suspicious Sysmon telemetry.
  • Unexpected outbound network connections.
  • Unauthorized processes executing.
  • Peppermint incident created automatically.

Initial Assessment

Determine whether containment is required immediately or whether additional evidence collection should occur before isolating the endpoint.

Review active network connections. 


netstat -ano

Review active processes. 


tasklist

Diagnostic Procedure

  • Identify affected endpoint.
  • Determine user currently logged in.
  • Identify suspicious processes.
  • Review Sysmon process activity.
  • Review Wazuh alert timeline.
  • Determine whether additional hosts are affected.

Root Cause Analysis

  • Malicious executable.
  • PowerShell abuse.
  • Credential compromise.
  • Malicious scheduled task.
  • User executed untrusted software.

Recovery Procedure

If compromise is confirmed, isolate the endpoint from the production VLAN while preserving forensic evidence. Avoid deleting files or reinstalling the operating system until evidence collection is complete.

Terminate confirmed malicious processes only after sufficient evidence has been collected. 


taskkill /PID <PID> /F

Continue monitoring Wazuh for additional alerts originating from other systems.

Validation

  • Compromised endpoint isolated.
  • No additional lateral movement detected.
  • Infrastructure services remain operational.
  • Evidence preserved.
  • Incident documentation updated.

Scenario 8 – Incident Escalation through n8n and Peppermint

The JCAG-LAB environment integrates Wazuh with n8n workflow automation and Peppermint IT Service Management to demonstrate automated incident handling. Once Wazuh generates a qualifying alert, n8n processes the event and creates an incident ticket for operational tracking.

Business Impact

  • Delayed incident response if automation fails.
  • Loss of operational visibility.
  • Incomplete incident documentation.
  • Security alerts may not be assigned.
  • SOC workflow interrupted.

Symptoms

  • Wazuh alert generated.
  • No n8n workflow execution.
  • No Peppermint ticket created.
  • Webhook errors observed.
  • Automation workflow failures.

Initial Assessment

Verify that the Wazuh alert exists.

Verify n8n workflow execution history.

Verify whether a Peppermint incident ticket was created.

Diagnostic Procedure

  • Review Wazuh alert details.
  • Review n8n execution logs.
  • Verify webhook connectivity.
  • Verify Peppermint API availability.
  • Confirm ticket creation status.

Verify Docker containers supporting automation services. 


docker ps

docker compose ps

Review container logs. 


docker logs n8n

docker logs peppermint

Root Cause Analysis

  • Webhook unavailable.
  • n8n workflow failure.
  • Peppermint API unavailable.
  • Docker container failure.
  • Authentication failure between services.

Recovery Procedure

  • Restore Docker services if necessary.
  • Restart failed containers.
  • Verify API communication.
  • Reprocess the security event.
  • Confirm incident ticket generation.

Validation

  • n8n workflow executed successfully.
  • Peppermint ticket created.
  • Alert successfully correlated.
  • Incident assigned for investigation.
  • Automation pipeline restored.

Key Takeaways

  • Security operations are most effective when detection, investigation, containment, recovery, and documentation follow a standardized operational workflow.
  • The JCAG-LAB security architecture integrates Windows Event Logs, Sysmon telemetry, Wazuh SIEM, n8n automation, and Peppermint ITSM into a complete enterprise-inspired incident response pipeline.

  • Every completed incident should improve future response capability through documentation, detection tuning, and operational lessons learned.

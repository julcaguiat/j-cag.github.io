Infrastructure Recovery & Troubleshooting

Playbook 6: Power, Cooling & Environmental Operations

Overview

Enterprise infrastructure availability depends not only on network services and operating systems, but also on reliable power delivery, proper environmental conditions, and healthy storage devices. Unexpected power interruptions, improper shutdowns, overheating, or storage degradation can affect multiple services simultaneously and may result in virtual machine corruption, data loss, or extended recovery operations.

This playbook documents the operational procedures used within the JCAG-LAB environment to recover from power-related incidents, validate hardware health, protect virtual machine storage, and safely restore enterprise services after an unexpected infrastructure event.

Unlike previous playbooks that focused primarily on software and network components, this playbook concentrates on the physical infrastructure supporting the virtual environment, including the Proxmox host, pfSense appliance, Cisco network equipment, storage tiers, and environmental operating conditions. Whenever possible, procedures are based on the deployed JCAG-LAB environment rather than generic enterprise examples.

Learning Objectives

  • Understand the physical power and environmental architecture of the JCAG-LAB environment.
  • Identify infrastructure risks associated with unexpected power loss and improper shutdowns.
  • Validate the health of storage devices supporting virtual machine workloads.
  • Perform structured recovery of infrastructure following environmental or power-related incidents.
  • Verify hardware health before returning enterprise services to production.

Storage Protection Strategy

The JCAG-LAB virtualization platform separates storage into three functional tiers to balance performance, capacity, and recoverability. Understanding these tiers is essential when performing recovery operations after hardware failures, unexpected shutdowns, or storage-related incidents.

Storage Tier Purpose Primary Workloads
Tier 0 – SSD (local-lvm) High-performance VM storage Windows Server 2019, Wazuh, n8n, Automation Services
Tier 1 – HDD-01 (local-lvm-hdd-01) General-purpose virtual machine storage Lab workloads and non-critical virtual machines
Tier 2 – HDD-02 (Directory Storage) Backup repository VZDump backups and recovery images

Storage Design Rules

  • Tier 0 provides high-performance storage for latency-sensitive virtual machines.
  • Tier 1 provides additional capacity using thin provisioning where appropriate.
  • Tier 2 stores recovery data and must remain isolated from production VM storage.
  • Snapshots provide rollback capability but do not replace backups.
  • Virtual machine backups should always be written to Tier 2 storage.

Following storage-related incidents, administrators should validate filesystem integrity, confirm virtual disk accessibility, and verify backup availability before attempting service restoration.

Scenario 1 – Unexpected Power Loss

An unexpected power outage is one of the highest-impact events that can affect the JCAG-LAB infrastructure. Because multiple enterprise services are hosted on a single Proxmox server, loss of power may simultaneously interrupt Active Directory, DNS, DHCP, Wazuh SIEM, n8n automation, Peppermint ITSM, Cisco Modeling Labs, and other virtualized services.

The objective of this scenario is to safely recover the infrastructure while minimizing the risk of filesystem corruption and virtual machine damage.

Business Impact

  • All virtual machines become unavailable.
  • Authentication services stop responding.
  • Network services become unavailable.
  • Monitoring and automation cease functioning.
  • Lab management access is lost.

Symptoms

  • Proxmox host powered off unexpectedly.
  • Virtual machines unavailable.
  • Users cannot access enterprise services.
  • Network monitoring stops receiving telemetry.
  • Management interfaces become unreachable.

Initial Assessment

Before restoring power, perform a physical inspection of the environment.

  • Verify commercial power has stabilized.
  • Inspect power cables.
  • Inspect storage devices for abnormal indicators.
  • Inspect switch and router status LEDs.
  • Confirm no burning smell or hardware damage.

Diagnostic Procedure

Once the Proxmox host boots successfully, verify storage availability. 


lsblk

df -h

pvesm status

Review recent system boot events. 


journalctl -b

journalctl -p err

Root Cause Analysis

  • Commercial power interruption.
  • Accidental power removal.
  • Power strip or UPS failure.
  • Improper shutdown sequence.
  • Electrical circuit interruption.

Recovery Procedure


qm list

systemctl status pve-cluster

systemctl status pvedaemon

systemctl status pveproxy

After confirming the Proxmox platform is healthy, restore virtual machines using the documented startup sequence described later in this playbook rather than starting all virtual machines simultaneously.

Validation

  • Proxmox host operational.
  • Storage pools available.
  • Critical virtual machines restored.
  • Network connectivity verified.
  • No storage errors detected.

Scenario 2 – Improper Shutdown Recovery

An improper shutdown occurs when the Proxmox host loses power before virtual machines complete a graceful shutdown. Although modern filesystems reduce the risk of corruption, administrators should always verify storage integrity before returning services to production.

Business Impact

  • Filesystem inconsistency.
  • Virtual machine corruption.
  • Database recovery required.
  • Container services fail to start.
  • Application instability.

Symptoms

  • Unexpected filesystem repair.
  • Virtual machines fail to start.
  • Docker services unavailable.
  • Database recovery messages.
  • Application startup failures.

Initial Assessment

journalctl -b -1

last -x

df -h

lsblk

dmesg | grep -i ext4

dmesg | grep -i xfs

Verify Proxmox storage status. 


pvesm status

Root Cause Analysis

  • Unexpected electrical outage.
  • Forced system shutdown.
  • Filesystem recovery after reboot.
  • Storage controller interruption.
  • Hardware instability.

Recovery Procedure

Start critical virtual machines individually while monitoring system logs for errors. 


qm start 101

qm start 105

qm start 106

qm start 102

After each virtual machine starts, verify that the guest operating system completes its normal startup sequence before proceeding to the next system.

Validation

  • No filesystem errors remain.
  • Virtual machines start normally.
  • Applications recover successfully.
  • Storage operates normally.
  • System logs contain no critical errors.

Scenario 4 – SSD Tier 0 Health Degradation

Tier 0 (local-lvm) provides high-performance storage for the most critical virtual machines within the JCAG-LAB environment, including Windows Server 2019, Wazuh SIEM, and n8n Automation. Because these workloads are latency-sensitive, degradation of the SSD storage pool can directly impact authentication, monitoring, and automation services.

Unlike sudden disk failures, SSD degradation typically presents as increasing I/O latency, SMART warnings, or media wear indicators before complete failure. Routine monitoring allows administrators to replace storage before production services are affected.

Business Impact

  • Critical virtual machines experience degraded performance.
  • Authentication services become slower.
  • Monitoring latency increases.
  • Automation workflows are delayed.
  • Potential virtual machine storage corruption.

Symptoms

  • High disk latency.
  • SMART health warnings.
  • Slow virtual machine response.
  • I/O timeout messages.
  • Storage-related kernel errors.

Initial Assessment

Verify the current status of all Proxmox storage pools. 


pvesm status

Review filesystem utilization. 


df -h

lsblk

Diagnostic Procedure

Review SMART health information for the SSD device. 


smartctl -a /dev/sda

Example output:


SMART overall-health self-assessment test result:
PASSED

Review kernel messages for storage-related errors. 


dmesg | grep -i nvme

dmesg | grep -i sata

dmesg | grep -i error

Root Cause Analysis

  • SSD wear approaching end-of-life.
  • I/O errors.
  • Controller communication problems.
  • Insufficient available storage.
  • Hardware degradation.

Recovery Procedure

If SMART indicates hardware degradation, immediately verify the availability of Tier 2 backups before scheduling storage replacement. Avoid performing additional write-intensive maintenance on a failing SSD. 


pvesm status

ls /var/lib/vz/dump/

qm list

Following storage replacement, restore affected virtual machines from validated backups if required.

Validation

  • SMART health verified.
  • No storage errors reported.
  • Critical virtual machines operating normally.
  • Tier 0 performance restored.
  • Backups confirmed current.

Scenario 6 – Tier 2 Backup Storage Unavailable

Tier 2 consists of dedicated directory storage used exclusively for VZDump virtual machine backups and recovery images. Because this storage contains the primary recovery data for the JCAG-LAB environment, its availability must be verified before any maintenance activity or break/fix exercise begins.

Unlike Tier 0 and Tier 1, Tier 2 should never host running virtual machines. Its sole purpose is to preserve recoverable backup data independent of production storage.

Business Impact

  • New backups cannot be created.
  • Recovery operations become impossible.
  • Maintenance activities should be postponed.
  • Disaster recovery readiness reduced.
  • Infrastructure risk increases.

Symptoms

  • Backup jobs fail.
  • Directory storage unavailable.
  • Backup repository not mounted.
  • Missing VZDump archives.
  • Storage permission errors.

Initial Assessment

Verify the backup storage configuration. 


pvesm status

Verify backup repository availability. 


ls -lh /var/lib/vz/dump/

Example output:


vzdump-qemu-101-2026_06_20.vma.zst

vzdump-qemu-105-2026_06_20.vma.zst

Diagnostic Procedure

Review mounted filesystems. 


mount

df -h

Review storage configuration. 


cat /etc/pve/storage.cfg

Review backup task history. 


journalctl | grep vzdump

Root Cause Analysis

  • Directory storage unavailable.
  • Filesystem not mounted.
  • Storage permissions incorrect.
  • Disk failure.
  • Backup repository accidentally deleted.

Recovery Procedure

Restore access to the backup repository before performing any additional maintenance. Confirm that recent VZDump archives remain readable and initiate a new backup to validate repository functionality. 


ls -lh /var/lib/vz/dump/

vzdump 101 \
--mode snapshot \
--compress zstd \
--storage backup-directory

Replace backup-directory with the storage identifier defined in the Proxmox storage configuration for Tier 2.

Validation

  • Tier 2 storage online.
  • Backup repository accessible.
  • Existing backup archives verified.
  • New VZDump backup completed successfully.
  • Recovery capability fully restored.

Scenario 7 – High Temperature / Cooling Validation

Although the JCAG-LAB environment operates on enterprise-inspired hardware within a home laboratory, thermal management remains critical for maintaining hardware reliability. Elevated operating temperatures may reduce component life, increase storage errors, cause processor throttling, and ultimately result in unexpected service interruptions.

Administrators should routinely inspect airflow, verify cooling fans, monitor system temperatures, and remove dust accumulation during scheduled maintenance windows.

Business Impact

  • Reduced hardware lifespan.
  • Processor thermal throttling.
  • Storage performance degradation.
  • Unexpected system shutdown.
  • Service instability.

Symptoms

  • High CPU temperature.
  • Cooling fans operating continuously.
  • System performance degradation.
  • Unexpected thermal warnings.
  • Automatic shutdown due to overheating.

Initial Assessment

Perform a physical inspection before investigating software.

  • Verify unobstructed airflow.
  • Inspect intake and exhaust vents.
  • Confirm all cooling fans are operational.
  • Check for excessive dust accumulation.
  • Verify room ventilation.

Diagnostic Procedure

Review hardware sensor information where available. 


sensors

lm-sensors

Review processor utilization. 


top

htop

Review kernel logs for thermal events. 


journalctl -k

dmesg | grep -i thermal

Root Cause Analysis

  • Restricted airflow.
  • Dust accumulation.
  • High ambient temperature.
  • Cooling fan failure.
  • Sustained processor utilization.

Recovery Procedure

  • Restore adequate airflow.
  • Clean ventilation openings.
  • Reduce workload if necessary.
  • Verify cooling fans.
  • Continue monitoring temperatures.

Validation

  • Temperatures within normal operating range.
  • No thermal events reported.
  • Processor operating normally.
  • Stable infrastructure performance.
  • Services fully operational.

Key Takeaways

  • Power and environmental incidents should always be addressed by validating physical infrastructure before restoring virtualized services.
  • The three-tier storage architecture supports both performance and recoverability; Tier 0 provides high-performance VM storage, Tier 1 provides capacity for general workloads, and Tier 2 preserves VZDump backups required for disaster recovery.
  • Routine hardware monitoring using SMART diagnostics, storage utilization reviews, and Proxmox health checks helps identify potential failures before they affect enterprise services.
  • Virtual machines should be restored in dependency order, beginning with Windows Server 2019 infrastructure services, followed by monitoring, automation, and supporting applications.
  • Comprehensive post-rerecovery validation ensures that the JCAG-LAB environment is stable, recoverable, and fully operational before normal operations resume.

Playbook Completion Outcome

Upon completion of this playbook: Capable of responding to power, environmental, and storage-related incidents affecting the JCAG-LAB environment. This includes recovering from unexpected power loss, validating Proxmox host integrity, monitoring hardware health, protecting the three-tier storage architecture, restoring enterprise services using the documented startup sequence, verifying infrastructure stability, and returning the virtualization platform and supporting enterprise services to full operational status using structured recovery procedures.

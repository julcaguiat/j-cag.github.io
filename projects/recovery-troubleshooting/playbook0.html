Playbook 0: Pre-Maintenance Safeguards & Recovery Procedures

Overview

Playbook 0 establishes the mandatory pre-maintenance procedures that must be completed before performing any maintenance activity, configuration change, software upgrade, operating system recovery, storage modification, failure simulation, or break/fix exercise within the JCAG-LAB enterprise environment.

Unlike deployment documentation, this playbook focuses on protecting existing services before work begins. The objective is to ensure every maintenance activity has an identified recovery path, validated backups, rollback capability, and documented baseline so infrastructure can be restored quickly if an unexpected failure occurs.

This playbook serves as the prerequisite for every recovery scenario contained within the Infrastructure Recovery & Troubleshooting project. Every subsequent playbook assumes these safeguards have already been completed before troubleshooting or maintenance activities begin.

This procedure applies to all enterprise systems deployed within the JCAG-LAB environment, including physical infrastructure, virtual infrastructure, storage resources, enterprise services, and monitoring platforms.

Enterprise Maintenance Philosophy

Enterprise maintenance is not simply making configuration changes. Every change introduces operational risk that must be controlled before work begins. A disciplined maintenance process reduces downtime, protects production services, and provides a repeatable recovery strategy if changes do not produce the expected result.

Enterprise Storage Architecture Verification

Before creating snapshots or performing maintenance, verify the storage architecture hosting the target virtual machine. Understanding where a workload resides determines which recovery methods are available and ensures backups are written to the appropriate storage tier.

The JCAG-LAB environment separates storage into three functional tiers to isolate production workloads, general-purpose virtual machines, and recovery data.

Storage Tier Storage Name Purpose
Tier 0 local-lvm (SSD) Performance-critical virtual machines
Tier 1 local-lvm-hdd-01 General-purpose virtual machine storage
Tier 2 HDD-02 Directory Storage VZDump backups and recovery archives
Why this matters

One of the most common misconceptions in virtualization environments is treating snapshots as backups. Snapshots provide a short-term rollback point but remain dependent on the original virtual disk. A hardware failure affecting the storage device can destroy both the virtual machine and its snapshots. Recovery backups must therefore reside on an independent storage tier.

Enterprise Storage Design Rules

The following operational standards apply throughout every recovery playbook within this project.

Rule Description
Snapshots are not backups Snapshots provide rollback capability only.
Tier 2 stores backups All VZDump backup archives must reside on the dedicated backup repository.
Tier 0 priority Reserved for performance-sensitive enterprise services.
Tier 1 capacity Supports general-purpose workloads and thin provisioning.
Separate recovery storage Backup repositories must remain isolated from production VM storage.
Backup verification required Every maintenance activity requires verification of a recoverable backup.

Create Recovery Snapshot

Once a verified backup exists on the Tier 2 backup repository, create a recovery snapshot of the target virtual machine. The snapshot captures the current VM state and provides a rapid rollback point if the maintenance activity introduces an unexpected issue.

Snapshots should be treated as temporary maintenance checkpoints rather than long-term backup solutions. They should be retained only for the duration of the maintenance activity unless organizational policy specifies otherwise.

Why this matters

Creating a snapshot immediately before maintenance minimizes rollback time. If a configuration change fails, the virtual machine can often be restored within minutes instead of performing a complete VZDump restoration.

Maintenance Risk Assessment

Before implementing any maintenance activity, evaluate the potential impact on infrastructure services. Risk assessment determines whether the planned change can proceed safely, whether additional safeguards are required, or whether the maintenance window should be postponed.

The objective is not to eliminate all risk, but to understand the potential consequences and ensure an appropriate recovery strategy has been established before changes are introduced.

Risk Category Examples Potential Impact
Configuration Changes Network settings, Active Directory, DNS, DHCP Authentication or connectivity failures
Operating System Updates Windows Updates, Linux package upgrades Boot failures or service startup issues
Storage Operations Disk expansion, VM migration, storage relocation VM corruption or storage exhaustion
Application Maintenance Docker services, Peppermint, n8n, Wazuh Application outage
Network Maintenance Firewall, VLAN, routing changes Loss of management or production connectivity
Why this matters

Most enterprise outages are not caused by hardware failures—they are introduced during planned maintenance. A structured risk assessment minimizes unexpected downtime and ensures recovery procedures have already been planned before implementation begins.

Maintenance Classification

Every maintenance activity should be categorized according to its operational risk. This classification determines the level of validation and recovery preparation required.

Classification Description Examples
Low Risk Minimal service impact Documentation updates, log review, configuration backup
Moderate Risk Limited interruption possible Software updates, service restart, VM configuration changes
High Risk Potential service interruption Storage migration, firewall changes, Active Directory modifications
Critical Risk Infrastructure-wide impact Hypervisor maintenance, storage replacement, network core modifications

Pre-Maintenance Validation Questions

The administrator should answer the following questions before approving the maintenance activity.

  • Has the correct maintenance window been scheduled?
  • Has the affected infrastructure been identified?
  • Are current backups available on Tier 2 storage?
  • Has a recovery snapshot been created?
  • Can the maintenance be reversed if necessary?
  • Has the expected business impact been documented?
  • Have dependent services been identified?
  • Is sufficient storage capacity available?
  • Have stakeholders been notified if required?
  • Has the rollback procedure been verified?

No-Go Conditions

Maintenance should be postponed immediately if any of the following conditions exist:

  • No verified Tier 2 backup is available.
  • The virtual machine is already experiencing operational issues.
  • Recovery procedures have not been documented.
  • Storage capacity is critically low.
  • The maintenance scope has changed significantly.
  • The administrator cannot verify the expected recovery path.

Emergency Rollback Procedure

If the maintenance activity introduces instability, restore the virtual machine to its pre-maintenance snapshot before attempting additional troubleshooting.

Step 1 — Stop the Virtual Machine


qm stop 100

Example output:


stopping VM 100...
status: stopped

Step 2 — Restore the Recovery Snapshot


qm rollback 100 pre-maintenance

Example output:


Restoring snapshot...

Rollback completed successfully.

Step 3 — Start the Virtual Machine


qm start 100

Example output:


starting VM 100...

status: running
Why this matters

Snapshot rollback provides the fastest method for recovering from unsuccessful maintenance. Because the snapshot was created immediately before the change, the virtual machine returns to its previously validated operational state with minimal downtime.

Key Takeaways

  • Every maintenance activity begins with risk reduction rather than technical changes.
  • Snapshots provide rapid rollback capability but must never replace proper backups.
  • Tier 2 backup storage is the authoritative recovery source for catastrophic VM failures.
  • Verifying VM health before maintenance prevents troubleshooting unrelated pre-existing issues.
  • Recovery planning is part of every maintenance activity, not something performed only after failures occur.
  • Documentation, change tracking, and validation are fundamental operational disciplines in enterprise infrastructure.
  • Following a standardized pre-maintenance procedure greatly reduces recovery time during future break/fix scenarios.

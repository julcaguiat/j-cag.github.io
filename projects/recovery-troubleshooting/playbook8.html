Infrastructure Recovery & Troubleshooting

Playbook 8: Configuration Integrity

Overview

Disaster recovery is the coordinated process of restoring critical infrastructure, enterprise services, and operational capabilities following a major disruption. Unlike the previous playbooks, which focus on recovering individual components or service domains, this playbook addresses recovery of the entire JCAG-LAB environment using documented procedures, validated backups, and structured restoration workflows.

Within the JCAG-LAB environment, disaster recovery encompasses the restoration of the Proxmox virtualization platform, three-tier storage architecture, Windows Server 2019 infrastructure services, Windows endpoints, Cisco network infrastructure, Wazuh SIEM, Sysmon telemetry, n8n automation, Peppermint IT Service Management, and supporting enterprise applications. The objective is to return the environment to a known-good operational state while preserving data integrity and minimizing recovery time.

This playbook consolidates the recovery methodologies developed throughout the previous playbooks into a unified disaster recovery strategy. Administrators will follow a structured sequence that prioritizes infrastructure dependencies, validates backup integrity, restores enterprise services, and confirms that the entire operational environment has returned to normal operation.

Learning Objectives

  • Understand the disaster recovery architecture implemented within the JCAG-LAB environment.
  • Prioritize infrastructure recovery based on service dependencies and business impact.
  • Restore enterprise services using validated backups and documented recovery procedures.
  • Verify operational integrity following complete infrastructure restoration.
  • Apply structured disaster recovery practices consistent with enterprise data center operations.

The disaster recovery design incorporates three functional storage tiers:

Storage Tier Purpose
Tier 0 – SSD (local-lvm) Performance-critical virtual machines including Windows Server 2019, Wazuh, and automation services.
Tier 1 – HDD (local-lvm-hdd-01) General-purpose virtual machines, laboratory workloads, and non-critical services.
Tier 2 – HDD Backup Directory Storage VZDump backups, restore images, and disaster recovery data.

Recovery operations assume that Tier 2 remains isolated from production virtual machine storage and serves as the authoritative source for backup restoration. Snapshots may accelerate operational recovery but are not considered a replacement for validated backups.

Business Continuity Philosophy

Business continuity focuses on maintaining or restoring essential operational capabilities following a disruptive event. Within the JCAG-LAB environment, business continuity emphasizes restoring core infrastructure services first so that dependent systems can be recovered in an orderly and predictable manner.

Recovery priorities are driven by service dependencies rather than convenience. Infrastructure services such as Active Directory, DNS, DHCP, storage, and virtualization provide the foundation required by monitoring, automation, and incident management platforms.

The business continuity philosophy adopted throughout the JCAG-LAB environment is based on the following principles:

  • Protect personnel before recovering equipment.
  • Preserve data integrity before restoring services.
  • Restore foundational infrastructure before dependent applications.
  • Validate every recovered service before proceeding to the next recovery stage.
  • Document every recovery action to support future operational improvement.
  • Regularly validate backups through controlled restore testing.

This approach ensures that disaster recovery activities remain repeatable, measurable, and consistent with enterprise operational practices.

Recovery Priorities

Not every system should be restored simultaneously. Administrators should restore services according to operational dependency and overall impact to the environment. The following recovery priorities define the recommended order for restoring the JCAG-LAB infrastructure.

Priority Infrastructure Component Reason for Priority
1 Physical Infrastructure Provides stable power, cooling, and hardware readiness.
2 Tier 0, Tier 1, and Tier 2 Storage Required before virtualization services can be restored.
3 Proxmox Virtual Environment Hosts all enterprise virtual machines.
4 Windows Server 2019 (AD DS, DNS, DHCP) Provides authentication, name resolution, and IP address management.
5 Wazuh Manager Restores centralized monitoring and security visibility.
6 n8n Automation Restores automated operational workflows.
7 Peppermint ITSM Restores incident tracking and operational management.
8 Cisco Modeling Labs Restores network simulation services.
9 Windows 11 Endpoints Reconnects client systems after core infrastructure is available.
10 Operational Validation Confirms complete restoration of enterprise services.

Following this sequence minimizes cascading failures and reduces the likelihood of recovering dependent services before prerequisite infrastructure has been validated.

Scenario 1 – Complete Proxmox Host Failure

A complete Proxmox host failure represents one of the highest-impact disaster recovery events within the JCAG-LAB environment. Since the Proxmox host supports Windows Server 2019, Wazuh Manager, n8n, Peppermint ITSM, Cisco Modeling Labs (CML), and other enterprise workloads, failure of the virtualization platform results in the loss of multiple critical services.

Recovery focuses on restoring the Proxmox host, validating storage integrity, and recovering virtual machines from the three-tier storage architecture or Tier 2 backup repository.

Business Impact

  • All hosted virtual machines become unavailable.
  • Authentication, DNS, DHCP, monitoring, automation, and ticketing services are interrupted.
  • Enterprise administration cannot be performed.
  • Security visibility is lost until monitoring services are restored.
  • Complete infrastructure outage.

Symptoms

  • Proxmox web interface unavailable.
  • Virtual machines inaccessible.
  • No response from management IP (10.10.100.10).
  • Storage pools unavailable.
  • Enterprise services unreachable.

Initial Assessment

Verify that the failure is isolated to the Proxmox host rather than the surrounding network infrastructure. 


ping 10.10.100.10

If the host is accessible through the console, verify the current system status. 


hostnamectl

uptime

systemctl status pvedaemon

systemctl status pveproxy

systemctl status pve-cluster

Diagnostic Procedure

  • Inspect server hardware for POST errors.
  • Verify BIOS detects all storage devices.
  • Confirm Tier 0 SSD and Tier 1 HDD are visible.
  • Validate Tier 2 backup storage availability.
  • Review Proxmox service status.
  • Determine whether restoration from backup is required.

Root Cause Analysis

  • Hardware failure.
  • Operating system corruption.
  • Storage subsystem failure.
  • Bootloader corruption.
  • Power interruption.
  • Administrative configuration error.

Recovery Procedure

If the Proxmox operating system cannot be recovered, reinstall Proxmox VE on replacement hardware or repaired storage, reconnect the existing storage pools, and restore virtual machines from Tier 2 VZDump backups when necessary. 


pvesm status

qmrestore /mnt/pve/backup/vzdump-qemu-101.vma.zst 101

qm list

Validation

  • Proxmox web interface available.
  • Storage pools online.
  • Virtual machines restored.
  • Enterprise services operational.
  • Recovery documentation completed.

Scenario 2 – Tier 0 SSD Failure

Tier 0 (local-lvm) provides high-performance storage for the most critical virtual machines within the JCAG-LAB environment, including Windows Server 2019, Wazuh Manager, and automation services. Failure of this storage tier directly affects enterprise authentication, monitoring, and infrastructure management.

Recovery should prioritize preserving available data, validating backup integrity, and restoring affected virtual machines to replacement SSD storage using Tier 2 backup media.

Business Impact

  • Critical virtual machines unavailable.
  • Authentication services interrupted.
  • Security monitoring unavailable.
  • Automation workflows halted.
  • Potential data loss if backups are unavailable.

Symptoms

  • Tier 0 storage missing from Proxmox.
  • Virtual machines fail to start.
  • I/O errors reported.
  • SMART failures detected.
  • Storage health alerts generated.

Initial Assessment

Verify storage visibility. 


pvesm status

lsblk

df -h

Review SMART diagnostics. 


smartctl -a /dev/sda

Diagnostic Procedure

  • Determine whether the SSD is detected by the operating system.
  • Review SMART health indicators.
  • Identify affected virtual machines.
  • Verify Tier 2 backup availability.
  • Confirm replacement storage availability.

Root Cause Analysis

  • SSD wear-out.
  • Controller failure.
  • Power-related corruption.
  • Filesystem corruption.
  • Hardware failure.

Recovery Procedure

Replace the failed SSD, recreate the Tier 0 storage pool, and restore critical virtual machines from Tier 2 VZDump backups. 


pvesm add lvmthin local-lvm

qmrestore /mnt/pve/backup/vzdump-qemu-101.vma.zst 101

Validation

  • Tier 0 storage restored.
  • Critical virtual machines operational.
  • Windows Server functioning normally.
  • Wazuh monitoring restored.
  • Backups remain available.

Scenario 3 – Active Directory Disaster Recovery

Active Directory provides centralized authentication, authorization, DNS, and directory services throughout the JCAG-LAB environment. Loss of the domain controller affects nearly every enterprise workload because user authentication, Group Policy processing, DNS resolution, and DHCP services depend on Windows Server 2019.

Disaster recovery focuses on restoring the domain controller from validated backups while preserving directory integrity and ensuring enterprise services become available in the proper sequence.

Business Impact

  • Domain authentication unavailable.
  • DNS services interrupted.
  • DHCP services unavailable.
  • Group Policy processing fails.
  • Domain-joined endpoints unable to authenticate.

Symptoms

  • Users cannot log in to the domain.
  • DNS name resolution fails.
  • DHCP leases unavailable.
  • Domain controller offline.
  • Enterprise applications cannot authenticate.

Initial Assessment

Verify that the domain controller virtual machine is running. 


qm list

Verify Windows Server availability. 


ping 10.10.100.11

Review Active Directory services. 


Get-Service NTDS

Get-Service DNS

Get-Service DHCPServer

Diagnostic Procedure

  • Determine whether the virtual machine boots successfully.
  • Review Windows Event Logs.
  • Verify Active Directory database integrity.
  • Validate DNS functionality.
  • Verify DHCP service availability.
  • Confirm client authentication.

Root Cause Analysis

  • Operating system corruption.
  • Active Directory database corruption.
  • Storage failure.
  • Administrative error.
  • Virtual machine corruption.

Recovery Procedure

Restore the Windows Server 2019 virtual machine from the most recent validated Tier 2 VZDump backup if service recovery is unsuccessful. After restoration, verify Active Directory, DNS, DHCP, and domain authentication before restoring dependent services such as Wazuh, n8n, and Peppermint. 


dcdiag

repadmin /replsummary

ipconfig /registerdns

Validation

  • Domain controller operational.
  • Active Directory healthy.
  • DNS resolution functioning.
  • DHCP leasing operational.
  • Domain authentication restored.
  • Dependent enterprise services functioning normally.

Scenario 4 – Wazuh SIEM Complete Recovery

The Wazuh Manager provides centralized security monitoring, log correlation, detection engineering, and alert generation for the JCAG-LAB environment. Failure of the Wazuh virtual machine eliminates centralized visibility into endpoint activity and prevents administrators from monitoring the security posture of the infrastructure.

Recovery procedures focus on restoring the Wazuh virtual machine, validating the integrity of the Wazuh Manager services, confirming agent connectivity, and ensuring that telemetry from Windows endpoints is once again collected, correlated, and presented within the Wazuh Dashboard.

Business Impact

  • Centralized security monitoring unavailable.
  • Endpoint telemetry no longer collected.
  • Detection engineering disabled.
  • Security alert generation interrupted.
  • Reduced incident response capability.

Symptoms

  • Wazuh Dashboard unavailable.
  • Agents reported offline.
  • No new alerts generated.
  • API unavailable.
  • Security monitoring interrupted.

Initial Assessment

Verify that the Wazuh virtual machine is running within Proxmox. 


qm list

Verify network connectivity. 


ping 10.10.100.15

Review Wazuh service status. 


systemctl status wazuh-manager

Diagnostic Procedure

  • Verify virtual machine availability.
  • Confirm storage availability.
  • Review Wazuh service logs.
  • Verify agent registration.
  • Confirm Dashboard accessibility.
  • Validate alert generation.

Root Cause Analysis

  • Virtual machine corruption.
  • Operating system failure.
  • Storage failure.
  • Configuration corruption.
  • Administrative error.

Recovery Procedure

Restore the Wazuh virtual machine from the latest validated Tier 2 VZDump backup if recovery of the existing virtual machine is unsuccessful. 


qmrestore /mnt/pve/backup/vzdump-qemu-105.vma.zst 105

systemctl restart wazuh-manager

Confirm that Windows endpoints reconnect automatically after the manager is restored.

Validation

  • Wazuh Dashboard operational.
  • Agents connected.
  • Alerts generated successfully.
  • Detection rules functioning.
  • Security monitoring restored.

Scenario 5 – n8n & Peppermint Recovery from Backup

The JCAG-LAB environment uses n8n workflow automation and Peppermint IT Service Management to automate security incident handling following alert generation by the Wazuh Manager. Recovery of these services ensures that automated workflows, API integrations, and incident tracking resume normal operation after a disaster event.

Both services are deployed as Docker-based workloads and should be restored only after the Proxmox host, storage infrastructure, Windows Server 2019, and Wazuh Manager have been successfully recovered.

Business Impact

  • Security automation unavailable.
  • Incident tickets not generated.
  • Workflow orchestration interrupted.
  • Reduced operational efficiency.
  • Manual incident handling required.

Symptoms

  • n8n web interface unavailable.
  • Peppermint inaccessible.
  • Docker containers stopped.
  • Webhook failures observed.
  • No incident tickets created.

Initial Assessment

Verify Docker service status. 


systemctl status docker

Review running containers. 


docker ps

docker compose ps

Diagnostic Procedure

  • Verify Docker daemon.
  • Confirm virtual machine health.
  • Inspect container logs.
  • Validate API connectivity.
  • Verify persistent storage availability.
  • Confirm backup integrity.

Review service logs. 


docker logs n8n

docker logs peppermint

Root Cause Analysis

  • Virtual machine corruption.
  • Docker service failure.
  • Persistent volume corruption.
  • Configuration error.
  • Database corruption.

Recovery Procedure

Restore the virtual machine or container data from the latest validated Tier 2 backup. Verify Docker functionality before starting application services. 


docker compose up -d

docker ps

Confirm successful communication between Wazuh, n8n, and Peppermint following service restoration.

Validation

  • Docker operational.
  • n8n workflows executing.
  • Peppermint accessible.
  • Incident tickets generated.
  • Automation pipeline restored.

Scenario 6 – Network Infrastructure Recovery Following Major Failure

Recovery of enterprise services depends upon restoration of the underlying network infrastructure. Following a major outage affecting routing, switching, or firewall services, administrators must restore the physical network before attempting to recover virtual machines and enterprise applications.

Within the JCAG-LAB environment, network recovery includes restoration of the Cisco ISR 1941 router, Cisco Catalyst 2960G switch, pfSense firewall, VLAN segmentation, Router-on-a-Stick configuration, management connectivity, and inter-VLAN routing.

Business Impact

  • Complete loss of network connectivity.
  • Virtual machines unreachable.
  • Authentication services inaccessible.
  • Monitoring platform disconnected.
  • Infrastructure administration interrupted.

Symptoms

  • Gateway unreachable.
  • Inter-VLAN routing unavailable.
  • DHCP clients fail to obtain addresses.
  • DNS resolution unavailable.
  • Management interfaces inaccessible.

Initial Assessment

Verify physical infrastructure.

  • Power supplied to network devices.
  • Switch uplinks connected.
  • Router operational.
  • pfSense online.
  • Management workstation connected.

Verify network connectivity. 


ping 10.10.255.1

ping 10.10.255.2

Diagnostic Procedure

Validate the Cisco router configuration. 


show ip interface brief

show ip route

show running-config

Validate switch operation. 


show vlan brief

show interfaces trunk

show spanning-tree

Validate pfSense connectivity and routing between the WAN segment (10.10.255.0/24) and internal VLANs before restoring enterprise applications.

Root Cause Analysis

  • Router failure.
  • Switch hardware failure.
  • Incorrect VLAN configuration.
  • Trunk failure.
  • Firewall configuration error.
  • Physical cabling issue.

Recovery Procedure

  • Restore pfSense connectivity.
  • Recover Cisco router configuration.
  • Restore VLANs.
  • Verify Router-on-a-Stick operation.
  • Validate inter-VLAN routing.
  • Restore management connectivity.
  • Reconnect Proxmox and enterprise virtual machines.

Validation

  • All VLANs operational.
  • Gateway reachable.
  • Inter-VLAN routing restored.
  • pfSense functioning normally.
  • Windows Server accessible.
  • Wazuh reachable.
  • Automation services operational.
  • Complete network infrastructure restored.

Scenario 8 – Backup Validation & Restore Testing

Backups are valuable only if they can be successfully restored. Routine backup validation confirms that recovery points stored on Tier 2 Backup Storage remain usable and that recovery procedures can be executed within acceptable recovery objectives.

The JCAG-LAB storage architecture separates production virtual machine storage from backup storage to protect recovery data from failures affecting Tier 0 SSD or Tier 1 HDD storage.

Business Impact

  • Undetected backup corruption.
  • Failed disaster recovery operations.
  • Extended recovery time.
  • Potential data loss.
  • Reduced confidence in recovery procedures.

Initial Assessment

Verify Tier 2 backup storage availability. 


pvesm status

df -h

List available virtual machine backups. 


ls -lh /mnt/pve/backup

Diagnostic Procedure

  • Verify backup timestamps.
  • Confirm backup file integrity.
  • Select a representative virtual machine for testing.
  • Perform a controlled restore into an isolated environment.
  • Validate application functionality after restoration.

Restore a virtual machine from a VZDump backup. 


qmrestore \
/mnt/pve/backup/vzdump-qemu-101.vma.zst \
201

After validation, remove the temporary recovery virtual machine if it is no longer required. 


qm stop 201

qm destroy 201

Root Cause Analysis

  • Backup corruption.
  • Storage media failure.
  • Incomplete backup job.
  • Human error.
  • Improper backup configuration.

Validation

  • Backup restored successfully.
  • Virtual machine booted normally.
  • Enterprise services operational.
  • Recovery documentation updated.
  • Backup integrity confirmed.

Key Takeaways

  • Successful disaster recovery depends on restoring infrastructure according to documented service dependencies rather than recovering systems in an arbitrary order.
  • The JCAG-LAB three-tier storage architecture separates production workloads from recovery data, ensuring Tier 2 backup storage remains the authoritative recovery source.
  • Snapshots improve operational flexibility but are not substitutes for validated backups stored on isolated backup media.
  • Enterprise recovery is complete only after virtualization, networking, storage, authentication, monitoring, automation, and endpoint services have all been functionally validated.
  • Regular disaster recovery exercises, backup validation, and documentation reviews improve operational readiness and reduce recovery time during real-world incidents.

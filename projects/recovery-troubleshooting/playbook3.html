Playbook 3: Network Infrastructure & Connectivity Recovery

Overview

This playbook documents the operational procedures used to diagnose and recover network infrastructure failures within the deployed JCAG-LAB enterprise environment. The procedures focus on Layer 1 through Layer 3 connectivity, providing a structured methodology for identifying network outages, isolating fault domains, restoring services, and validating operational functionality.

Unlike the previous playbook, which focused on physical infrastructure and structured cabling, this playbook begins once physical connectivity has been verified. The emphasis shifts toward logical network operation including VLAN segmentation, inter-VLAN routing, gateway accessibility, DHCP services, DNS resolution, management connectivity, and WAN access through the pfSense firewall.

All scenarios are performed against the production-style JCAG-LAB infrastructure whenever practical. Failures are introduced in a controlled manner and are immediately recoverable through documented operational procedures.

Learning Objectives

  • Verify Layer 1 through Layer 3 connectivity using structured troubleshooting procedures.
  • Diagnose VLAN, trunk, and inter-VLAN routing failures.
  • Validate DHCP, DNS, and default gateway operation across multiple VLANs.
  • Restore network connectivity following configuration or infrastructure failures.
  • Document recovery activities using standardized operational procedures.

JCAG-LAB Network Architecture

The enterprise network deployed within JCAG-LAB follows a segmented campus design consisting of dedicated user, server, and management networks connected through Router-on-a-Stick routing implemented on a Cisco ISR 1941 router. Scenario 1 – Default Gateway Unreachable

A workstation reports that local network resources remain accessible, but devices outside the local subnet cannot be reached. Internet access is unavailable and communication with systems located on other VLANs fails. Within the JCAG-LAB environment, this typically indicates a problem with the configured default gateway, Router-on-a-Stick subinterfaces, switch VLAN assignment, or the physical uplink between the Cisco Catalyst 2960G and the Cisco ISR 1941 router.

Business Impact

  • Loss of inter-VLAN communication.
  • Internet connectivity unavailable.
  • Access to enterprise services on VLAN 100 interrupted.
  • Remote management functions unavailable.

Symptoms

  • Local devices respond normally.
  • Unable to reach 10.10.100.11 (Windows Server).
  • Unable to reach pfSense gateway.
  • Internet connectivity unavailable.

Initial Assessment

Determine whether the issue affects a single endpoint, an entire VLAN, or all network segments. Confirm whether other hosts within VLAN 10 experience the same behavior before proceeding.

Diagnostic Procedure

Verify endpoint addressing. 


ipconfig /all

Example output: 


IPv4 Address . . . . . . : 10.10.10.25
Subnet Mask . . . . . .  : 255.255.255.0
Default Gateway . . . .  : 10.10.10.1
DNS Server . . . . . . . : 10.10.100.11

Test gateway connectivity. 


ping 10.10.10.1

Verify router interfaces. 


show ip interface brief

Example output: 


GigabitEthernet0/1.10    up    up
GigabitEthernet0/1.30    up    up
GigabitEthernet0/1.99    up    up
GigabitEthernet0/1.100   up    up

Root Cause Analysis

  • Incorrect default gateway configured.
  • Router subinterface administratively down.
  • Incorrect native VLAN.
  • Trunk failure between switch and router.
  • Switch access port assigned to incorrect VLAN.

Recovery Procedure

Correct endpoint gateway if necessary.

Verify router subinterfaces. 


show running-config interface g0/1.10

show running-config interface g0/1.30

show running-config interface g0/1.100

Verify switch trunk. 


show interfaces trunk

Validate end-to-end connectivity. 


ping 10.10.100.11

ping 10.10.255.1

Scenario 2 – VLAN Communication Failure

Endpoints connected to the Cisco Catalyst 2960G are unable to communicate with other devices assigned to the same VLAN. The failure is isolated to a specific broadcast domain while other VLANs continue operating normally.

Business Impact

  • Departmental communications interrupted.
  • File services unavailable.
  • Authentication requests fail.
  • Reduced productivity for affected users.

Symptoms

  • Hosts receive valid IP addresses.
  • Unable to ping neighboring systems on the same VLAN.
  • Switch interfaces operational.
  • No inter-VLAN failures observed.

Diagnostic Procedure

Verify VLAN database. 


show vlan brief

Example output: 


VLAN Name           Status

10   Sales          active

30   IT             active

99   Management     active

100  Servers        active

Verify switch interface assignment. 


show interfaces switchport

Inspect MAC address learning. 


show mac address-table dynamic

Verify interface status. 


show interfaces status

Root Cause Analysis

  • Incorrect access VLAN assignment.
  • Interface shutdown.
  • Duplicate VLAN configuration.
  • MAC address learning failure.
  • Incorrect patch panel termination.

Recovery Procedure

Assign the correct VLAN. 


configure terminal

interface fa0/10

switchport mode access

switchport access vlan 10

no shutdown

end

Verify operation. 


show vlan brief

show interfaces status

Scenario 3 – Inter-VLAN Routing Failure

Devices communicate successfully within their local VLAN but are unable to reach systems located in other VLANs. In the JCAG-LAB environment this usually indicates a Router-on-a-Stick configuration issue on the Cisco ISR 1941 or an 802.1Q trunk failure between the router and the Catalyst 2960G switch.

Business Impact

  • Users cannot access Windows Server services.
  • Administrative systems become unreachable.
  • Monitoring platforms cannot collect endpoint telemetry.
  • Enterprise applications become unavailable.

Symptoms

  • Gateway reachable.
  • Local VLAN communication functional.
  • Cross-VLAN communication fails.
  • Windows Server inaccessible from user VLANs.

Diagnostic Procedure

Verify routing table. 


show ip route

Verify subinterfaces. 


show running-config interface g0/1.10

show running-config interface g0/1.30

show running-config interface g0/1.99

show running-config interface g0/1.100

Verify encapsulation. 


show running-config | section interface GigabitEthernet0/1

Example output: 


interface GigabitEthernet0/1.10
 encapsulation dot1Q 10
 ip address 10.10.10.1 255.255.255.0

interface GigabitEthernet0/1.30
 encapsulation dot1Q 30
 ip address 10.10.30.1 255.255.255.0

interface GigabitEthernet0/1.99
 encapsulation dot1Q 99
 ip address 10.10.99.1 255.255.255.0

interface GigabitEthernet0/1.100
 encapsulation dot1Q 100
 ip address 10.10.100.1 255.255.255.0

Root Cause Analysis

  • Missing router subinterface.
  • Incorrect VLAN encapsulation.
  • IP addressing mismatch.
  • Trunk configuration failure.
  • Native VLAN mismatch.

Recovery Procedure

Verify trunk operation. 


show interfaces trunk

Verify routing. 


show ip route

Validate communication. 


ping 10.10.100.11

ping 10.10.30.21

tracert 10.10.100.11

Successful completion confirms that Layer 3 routing between VLAN 10, VLAN 30, VLAN 99, and VLAN 100 has been restored and enterprise services hosted on the server network are once again reachable.

Scenario 4 – Trunk Link Failure

A failure of the 802.1Q trunk between the Cisco Catalyst 2960G and Cisco ISR 1941 disrupts communication between VLANs by preventing tagged traffic from reaching the Router-on-a-Stick subinterfaces. In the JCAG-LAB environment, this typically affects every user VLAN simultaneously while individual access ports continue to function within their local broadcast domains.

Business Impact

  • Inter-VLAN communication unavailable.
  • Windows Server 2019 services become inaccessible from user VLANs.
  • DHCP relay and DNS services may appear unavailable.
  • Management access to enterprise systems is interrupted.

Symptoms

  • Endpoints communicate within their local VLAN.
  • Gateway cannot be reached from multiple VLANs.
  • Cross-VLAN traffic fails.
  • Server network becomes unreachable.

Diagnostic Procedure

Verify trunk status on the Catalyst 2960G. 


show interfaces trunk

Example output:


Port        Mode         Encapsulation  Status

Gi0/1       on           802.1Q         trunking

Allowed VLANs

10,30,99,100

Verify switch interface configuration. 


show running-config interface gigabitEthernet0/1

Verify router subinterfaces. 


show ip interface brief

Root Cause Analysis

  • Trunk port administratively down.
  • Incorrect encapsulation.
  • Allowed VLAN list modified.
  • Native VLAN mismatch.
  • Physical uplink failure.

Recovery Procedure


configure terminal

interface gigabitEthernet0/1

switchport mode trunk

switchport trunk allowed vlan 10,30,99,100

no shutdown

end

Validate trunk operation. 


show interfaces trunk

show spanning-tree interface gigabitEthernet0/1

Scenario 5 – Access Port VLAN Misconfiguration

An endpoint is connected to a functional switch port but is assigned to the incorrect VLAN. Although the physical connection remains operational, the device cannot communicate with its intended network resources because it has been placed into the wrong broadcast domain.

Business Impact

  • User cannot access departmental resources.
  • Authentication against Active Directory may fail.
  • File shares become unavailable.
  • Network segmentation policies are violated.

Symptoms

  • Switch port reports "connected".
  • Endpoint receives unexpected IP address.
  • Gateway belongs to another VLAN.
  • Unable to reach intended services.

Diagnostic Procedure

Verify endpoint configuration. 


ipconfig /all

Verify switchport configuration. 


show interfaces switchport

Verify VLAN membership. 


show vlan brief

Example output:


FastEthernet0/12

Administrative Mode : static access

Access Mode VLAN : 30

Root Cause Analysis

  • Incorrect access VLAN assignment.
  • Port reused without updating configuration.
  • Documentation mismatch.
  • Patch panel connected to the wrong switch port.

Recovery Procedure


configure terminal

interface fastEthernet0/12

switchport mode access

switchport access vlan 10

description SALES-PC-01

no shutdown

end

Verify corrected configuration. 


show interfaces switchport

show vlan brief

Renew the client DHCP lease if necessary. 


ipconfig /release

ipconfig /renew

Scenario 6 – DHCP Service Failure Across VLANs

Endpoints fail to obtain IP addresses automatically from the Windows Server 2019 DHCP service. Depending on the failure location, this may affect a single VLAN or the entire enterprise network. In the JCAG-LAB environment, DHCP services are provided by Windows Server 2019 located within VLAN 100.

Business Impact

  • New devices cannot join the network.
  • Users receive Automatic Private IP Addresses (APIPA).
  • Authentication and enterprise services become unavailable.
  • Operational productivity is interrupted.

Symptoms

  • IP address begins with 169.254.x.x.
  • Unable to contact domain services.
  • Internet access unavailable.
  • Multiple endpoints affected simultaneously.

Diagnostic Procedure

Verify current IP configuration. 


ipconfig /all

Attempt DHCP renewal. 


ipconfig /release

ipconfig /renew

Verify DHCP relay configuration on the router. 


show running-config interface g0/1.10

show running-config interface g0/1.30

Example output:


ip helper-address 10.10.100.11

Verify Windows Server DHCP service. 


Get-Service DHCPServer

Verify DHCP scope utilization. 


Get-DhcpServerv4Scope

Get-DhcpServerv4Lease

Root Cause Analysis

  • DHCP service stopped.
  • Incorrect IP Helper configuration.
  • DHCP scope exhausted.
  • Firewall blocking DHCP traffic.
  • Network connectivity to Windows Server unavailable.

Recovery Procedure

Restart DHCP service if required. 


Restart-Service DHCPServer

Verify IP Helper configuration. 


show running-config interface g0/1.10

show running-config interface g0/1.30

Renew client address. 


ipconfig /release

ipconfig /renew

Validate successful lease assignment.

Example output:


IPv4 Address . . . . . . : 10.10.10.26

Default Gateway . . . .  : 10.10.10.1

DHCP Server . . . . . .  : 10.10.100.11

Lease Obtained . . . . . : Successful

Successful completion confirms that DHCP services are functioning correctly, clients receive valid addressing information, and enterprise connectivity has been restored across the affected VLANs.

Scenario 7 – DNS Resolution Failure

The Windows Server 2019 domain controller provides DNS services for the JCAG-LAB environment. When DNS becomes unavailable, endpoints may still have network connectivity but are unable to resolve hostnames, authenticate to the domain, access shared resources, or communicate with internal services using their Fully Qualified Domain Names (FQDNs).

Business Impact

  • Users cannot log into the Active Directory domain.
  • Network resources cannot be accessed by hostname.
  • Applications relying on DNS fail.
  • Internet browsing fails despite valid IP connectivity.

Symptoms

  • Successful ping to IP addresses but failure to resolve hostnames.
  • Unable to access jcag-lab.local resources.
  • Authentication delays.
  • Name resolution timeout errors.

Diagnostic Procedure

Verify current DNS configuration. 


ipconfig /all

Verify name resolution. 


nslookup jcag-lab.local

nslookup dc01.jcag-lab.local

Verify Windows DNS service. 


Get-Service DNS

Verify network connectivity to the domain controller. 


ping 10.10.100.11

Root Cause Analysis

  • DNS service stopped.
  • Incorrect DNS server assigned by DHCP.
  • Firewall blocking DNS traffic.
  • Network path interruption.
  • Corrupted DNS zone.

Recovery Procedure


Restart-Service DNS

Verify DNS functionality. 


Resolve-DnsName jcag-lab.local

Confirm successful client resolution.

Scenario 8 – pfSense WAN Connectivity Failure

The pfSense firewall provides WAN connectivity between the JCAG-LAB enterprise network and the home ISP router. Loss of WAN connectivity prevents Internet access while internal VLAN communication continues to function normally.

Business Impact

  • Internet unavailable.
  • Software updates fail.
  • External package repositories inaccessible.
  • Cloud-based services unavailable.

Symptoms

  • Internal communication remains operational.
  • Unable to reach public Internet addresses.
  • Gateway reachable but external destinations fail.
  • pfSense dashboard reports WAN down.

Diagnostic Procedure

Verify pfSense WAN interface status. 


Status → Interfaces

Verify WAN IP assignment. 


Interfaces → WAN

Verify routing table. 


Diagnostics → Routes

Verify Internet connectivity. 


ping 8.8.8.8

ping 1.1.1.1

Root Cause Analysis

  • WAN interface disconnected.
  • ISP gateway unavailable.
  • Incorrect default gateway.
  • NAT configuration error.
  • Firewall rule blocking outbound traffic.

Recovery Procedure

  • Verify WAN cable connectivity.
  • Renew WAN DHCP lease if applicable.
  • Confirm default gateway configuration.
  • Validate outbound NAT rules.
  • Confirm Internet connectivity after recovery.

Scenario 9 – Management VLAN Connectivity Failure

The Management VLAN (VLAN 99) provides administrative access to the Cisco Catalyst 2960G, Cisco ISR 1941, pfSense firewall, and enterprise infrastructure devices. Loss of management connectivity prevents administrative operations while production traffic may continue to operate normally.

Business Impact

  • Unable to manage infrastructure devices.
  • SSH sessions unavailable.
  • Configuration changes cannot be performed remotely.
  • Monitoring systems lose management visibility.

Symptoms

  • SSH connection refused.
  • Management IP unreachable.
  • Switch continues forwarding production traffic.
  • Infrastructure services remain operational.

Diagnostic Procedure


show ip interface brief


show vlan brief


show running-config interface vlan 99


show ssh

Root Cause Analysis

  • Incorrect management VLAN assignment.
  • Missing default gateway.
  • SSH disabled.
  • ACL blocking management access.
  • Management interface shutdown.

Recovery Procedure


configure terminal

interface vlan 99

ip address 10.10.99.2 255.255.255.0

no shutdown

exit

ip default-gateway 10.10.99.1

end

Verify administrative connectivity. 


show ip interface brief

show ssh

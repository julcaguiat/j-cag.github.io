Playbook 5: Monitoring, Logging & Alert Recovery

Overview

Playbook 5 focuses on monitoring, logging, alerting, and automation recovery within the JCAG-LAB enterprise infrastructure. The objective is to diagnose, restore, and validate the operational visibility provided by the monitoring platform following component failures, telemetry interruptions, or automation workflow issues.

Unlike previous playbooks that primarily restore operating systems or network services, this playbook concentrates on ensuring that infrastructure events are properly collected, correlated, processed, and converted into actionable incidents. Maintaining operational visibility is essential because infrastructure failures cannot be effectively investigated if monitoring systems themselves are degraded.

The monitoring pipeline deployed within JCAG-LAB consists of Windows endpoints generating Sysmon telemetry, Wazuh agents forwarding security events to the Wazuh Manager, automation workflows executed by n8n, and incident creation through the Peppermint ITSM platform.

Learning Objectives

  • Understand the monitoring architecture deployed within JCAG-LAB.
  • Diagnose failures affecting Wazuh, Sysmon, n8n, and Peppermint.
  • Validate end-to-end telemetry collection and alert generation.
  • Restore automation workflows following infrastructure failures.
  • Verify complete monitoring visibility before returning systems to production.

Scenario 1 – Wazuh Manager Unavailable

The Wazuh Manager is the central component of the JCAG-LAB monitoring platform. It receives telemetry from managed endpoints, performs rule evaluation, correlates security events, and generates alerts consumed by downstream automation workflows. If the manager becomes unavailable, endpoint monitoring continues locally, but centralized visibility and alert generation are lost.

Within the JCAG-LAB environment, the Wazuh Manager is deployed as a dedicated Ubuntu Server virtual machine hosted on Proxmox Tier 0 SSD storage.

Business Impact

  • Security events are no longer centrally collected.
  • Detection rules stop processing new telemetry.
  • Dashboards display stale or missing data.
  • n8n workflows are no longer triggered.
  • Peppermint incident creation stops.

Symptoms

  • Unable to access the Wazuh Dashboard.
  • Agents appear disconnected.
  • No new security alerts generated.
  • API requests fail.
  • Monitoring dashboards stop updating.

Initial Assessment

Verify that the virtual machine is operational before troubleshooting Wazuh services. 


qm status 105

ping 10.10.100.15

ssh user@10.10.100.15

Example output:


status: running
64 bytes from 10.10.100.15

Diagnostic Procedure

Verify the status of the Wazuh Manager service. 


sudo systemctl status wazuh-manager

Review recent service logs. 


sudo journalctl -u wazuh-manager -n 100

sudo tail -100 /var/ossec/logs/ossec.log

Verify listening ports. 


sudo ss -tulpn

Root Cause Analysis

  • Wazuh Manager service stopped.
  • Operating system resource exhaustion.
  • Disk capacity exceeded.
  • Configuration corruption.
  • Unexpected operating system reboot.

Recovery Procedure


sudo systemctl restart wazuh-manager

sudo systemctl status wazuh-manager

If the service fails to restart successfully, investigate configuration integrity before restoring the virtual machine using the approved snapshot or Tier 2 VZDump backup documented in Playbook 0.

Validation

  • Manager service running.
  • Dashboard accessible.
  • Agents reconnect successfully.
  • Alerts generated.
  • Automation resumes.

Scenario 2 – Wazuh Agent Offline

Each monitored endpoint forwards security telemetry through the Wazuh Agent. When an agent becomes unavailable, monitoring for that endpoint is interrupted, even though the Wazuh Manager continues operating normally.

Within JCAG-LAB, this scenario primarily applies to the Windows 11 workstation used for Sysmon telemetry generation and security monitoring validation.

Business Impact

  • Endpoint no longer monitored.
  • Security events not forwarded.
  • Host disappears from monitoring dashboards.
  • Incident detection becomes incomplete.
  • Endpoint compliance cannot be verified.

Symptoms

  • Agent shown as disconnected.
  • No recent events received.
  • Endpoint marked inactive.
  • Heartbeat timeout.
  • Security alerts stop for affected host.

Initial Assessment

Verify network connectivity between the endpoint and Wazuh Manager. 


ping 10.10.100.15

Test-NetConnection 10.10.100.15 -Port 1514

Test-NetConnection 10.10.100.15 -Port 1515

Diagnostic Procedure

Verify the Windows service. 


Get-Service WazuhSvc

Review agent logs. 


notepad "C:\Program Files (x86)\ossec-agent\ossec.log"

Confirm the manager recognizes the endpoint. 


sudo /var/ossec/bin/agent_control -l

Root Cause Analysis

  • Agent service stopped.
  • Network interruption.
  • Firewall blocking communication.
  • Authentication key mismatch.
  • Manager unavailable.

Recovery Procedure


Restart-Service WazuhSvc

If communication remains unavailable, verify agent registration, firewall configuration, and endpoint network connectivity before re-registering the agent.

Validation

  • Agent connected.
  • Endpoint appears online.
  • Heartbeat restored.
  • Events received.
  • Alerts generated successfully.

Scenario 3 – Sysmon Telemetry Collection Failure

Sysmon provides enhanced Windows telemetry used by Wazuh detection rules. Although Windows Event Logging may continue functioning, loss of Sysmon telemetry significantly reduces endpoint visibility and detection capability.

Within JCAG-LAB, Sysmon is installed on the Windows 11 endpoint used for attack simulation and detection engineering validation.

Business Impact

  • Process creation events unavailable.
  • Network connection monitoring reduced.
  • Registry monitoring interrupted.
  • Detection rules generate fewer alerts.
  • SOC visibility degraded.

Symptoms

  • No Sysmon events appear in Wazuh.
  • Sysmon Operational log stops updating.
  • Expected alerts no longer trigger.
  • Detection dashboards show reduced activity.

Initial Assessment

Verify that the Sysmon service is installed and running. 


Get-Service Sysmon64

Example output:


Status   Name
------   ----
Running  Sysmon64

Diagnostic Procedure

Review the Sysmon Operational log. 


Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" -MaxEvents 20

Verify configuration information. 


sysmon64.exe -c

Confirm that Wazuh continues receiving Sysmon events. 


Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" -MaxEvents 5

Root Cause Analysis

  • Sysmon service stopped.
  • Invalid configuration file.
  • Configuration update failed.
  • Event channel disabled.
  • Endpoint operating system failure.

Recovery Procedure


Restart-Service Sysmon64

sysmon64.exe -c

If configuration corruption is identified, redeploy the approved Sysmon configuration used throughout the JCAG-LAB environment and verify that the Operational log resumes generating events.

Validation

  • Sysmon service running.
  • Operational log updating.
  • Events forwarded to Wazuh.
  • Detection rules trigger successfully.
  • Monitoring pipeline fully restored.

Scenario 4 – Log Ingestion Failure

Log ingestion is the process of receiving endpoint telemetry, parsing security events, applying detection rules, and storing the resulting alerts for investigation. A disruption anywhere in this process can significantly reduce the Security Operations Center's visibility into enterprise activity.

Within the JCAG-LAB environment, log ingestion begins at the Windows endpoint, passes through the Wazuh Agent, and is processed by the Wazuh Manager before being presented in the Wazuh Dashboard and forwarded to automation workflows.

Business Impact

  • Security events no longer appear in Wazuh.
  • Detection rules stop generating alerts.
  • Dashboards display outdated information.
  • Automation workflows are never triggered.
  • Incident response visibility is lost.

Symptoms

  • No recent events displayed in Wazuh Dashboard.
  • Event counters stop increasing.
  • Agents appear connected but produce no telemetry.
  • Detection rules no longer trigger.
  • Security monitoring appears inactive.

Initial Assessment

Begin by confirming that endpoint telemetry is still being generated. 


Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" -MaxEvents 10

Verify that the Wazuh Agent remains connected. 


Get-Service WazuhSvc

Confirm the Wazuh Manager is operational. 


sudo systemctl status wazuh-manager

Diagnostic Procedure

Review Wazuh Manager logs for ingestion or decoding errors. 


sudo tail -100 /var/ossec/logs/ossec.log

sudo journalctl -u wazuh-manager

Verify available disk capacity. 


df -h

Review active agent status. 


sudo /var/ossec/bin/agent_control -l

Root Cause Analysis

  • Agent communication interrupted.
  • Manager service failure.
  • Disk storage exhausted.
  • Decoder or rules configuration error.
  • Event forwarding interruption.

Recovery Procedure


sudo systemctl restart wazuh-manager

sudo systemctl status wazuh-manager

If storage exhaustion is identified, free sufficient disk space before restarting the Wazuh Manager. Verify that new endpoint events immediately begin appearing within the dashboard after service restoration.

Validation

  • New endpoint events received.
  • Dashboard updates in real time.
  • Alerts generated successfully.
  • Agent communications restored.
  • Monitoring pipeline operational.

Scenario 5 – Docker Service Failure (Supporting Containers)

Several supporting services within JCAG-LAB are deployed using Docker, including Peppermint ITSM and associated application containers. A Docker engine or container failure may interrupt application availability while the underlying Ubuntu Server remains operational.

Although the Wazuh Manager itself is deployed as a virtual machine, supporting services within the monitoring pipeline may rely on Docker-based applications. This scenario focuses on recovering Docker services before restoring the entire operating system.

Business Impact

  • Peppermint unavailable.
  • Automation workflows fail.
  • API endpoints unreachable.
  • Containerized services offline.
  • Incident generation interrupted.

Symptoms

  • Docker containers stopped.
  • Web interface unavailable.
  • API connection failures.
  • Container restart loops.
  • Container exits unexpectedly.

Initial Assessment

Verify Docker Engine status. 


systemctl status docker

Review running containers. 


docker ps -a

Example output:


CONTAINER ID   STATUS
f8c2a31b       Exited (1)

Diagnostic Procedure

Review container logs. 


docker logs peppermint

docker logs postgres

Review Docker daemon logs. 


journalctl -u docker

Inspect container configuration. 


docker inspect peppermint

Root Cause Analysis

  • Docker Engine stopped.
  • Container configuration error.
  • Database dependency unavailable.
  • Storage volume corruption.
  • Resource exhaustion.

Recovery Procedure


sudo systemctl restart docker

docker compose up -d

docker ps

If containers continue failing after restart, verify the Docker Compose configuration, persistent volume mappings, and application logs before considering restoration from a validated Proxmox snapshot.

Validation

  • Docker Engine operational.
  • Required containers running.
  • Peppermint accessible.
  • Application logs normal.
  • API endpoints responding.

Scenario 6 – Security Alert Not Generated

One of the most difficult monitoring failures to detect occurs when telemetry is successfully collected but no alert is generated. This usually indicates a problem within the detection pipeline rather than the underlying infrastructure.

Within JCAG-LAB, administrators intentionally generate known security events to verify that Sysmon, the Wazuh Agent, the Wazuh Manager, detection rules, n8n, and Peppermint continue functioning as a complete end-to-end monitoring system.

Business Impact

  • Security incidents remain undetected.
  • Automation workflows never execute.
  • Incident tickets are not created.
  • SOC visibility becomes unreliable.
  • Compliance monitoring is reduced.

Symptoms

  • Expected alerts absent.
  • Telemetry visible but no correlated alert.
  • Automation workflow not triggered.
  • No Peppermint incident created.
  • Dashboard displays events without detections.

Initial Assessment

Verify that endpoint telemetry continues to reach the Wazuh Manager before investigating detection rules. 


Get-WinEvent -LogName "Microsoft-Windows-Sysmon/Operational" -MaxEvents 5

Confirm the endpoint remains connected. 


sudo /var/ossec/bin/agent_control -l

Diagnostic Procedure

Review recent Wazuh alerts. 


sudo tail -100 /var/ossec/logs/alerts/alerts.json

Verify that the expected detection rule is enabled. 


sudo grep "<rule" \
/var/ossec/etc/rules/*.xml

Review Wazuh Manager logs for rule processing errors. 


sudo tail -100 /var/ossec/logs/ossec.log

Root Cause Analysis

  • Detection rule disabled.
  • Rule logic incorrect.
  • Sysmon event not generated.
  • Telemetry decoder mismatch.
  • Manager configuration error.

Recovery Procedure

Correct the affected detection rule, reload the Wazuh Manager configuration, and repeat the controlled security event used for validation. 


sudo systemctl restart wazuh-manager

sudo systemctl status wazuh-manager

Confirm that the generated event produces a Wazuh alert, successfully triggers the downstream automation workflow, and ultimately creates an incident within Peppermint ITSM.

Validation

  • Detection rule executes successfully.
  • Security alert generated.
  • Alert appears in Wazuh Dashboard.
  • Automation workflow triggered.
  • Peppermint incident created.

Scenario 7 – n8n Workflow Failure

n8n serves as the automation platform within the JCAG-LAB environment. It receives alerts generated by Wazuh, executes predefined workflows, and communicates with downstream services such as Peppermint ITSM. A workflow failure interrupts automation even though the monitoring platform may continue to detect security events normally.

Because n8n is deployed as an Ubuntu Server virtual machine, troubleshooting should begin by verifying the operating system, followed by the n8n service, workflow configuration, webhook connectivity, and API communication.

Business Impact

  • Incident automation stops.
  • Security alerts require manual processing.
  • Peppermint tickets are not created automatically.
  • Notification workflows fail.
  • Operational response becomes delayed.

Symptoms

  • Workflow execution failures.
  • Webhook requests timeout.
  • Execution history shows errors.
  • API calls fail.
  • No automation occurs after Wazuh generates an alert.

Initial Assessment

Verify that the virtual machine hosting n8n is operational. 


qm status 106

ping 10.10.100.16

ssh user@10.10.100.16

Verify the n8n service. 


systemctl status n8n

Diagnostic Procedure

Review the n8n service logs. 


journalctl -u n8n -n 100

Verify webhook connectivity and workflow execution history from the n8n web interface.

Confirm network connectivity to both the Wazuh Manager and Peppermint server. 


ping 10.10.100.15

ping 10.10.100.12

Root Cause Analysis

  • n8n service stopped.
  • Workflow configuration error.
  • Webhook endpoint unavailable.
  • API authentication failure.
  • Network communication interrupted.

Recovery Procedure


sudo systemctl restart n8n

sudo systemctl status n8n

After restarting the service, manually trigger a known workflow and verify that it completes successfully before returning the automation platform to production.

Validation

  • n8n service operational.
  • Webhook reachable.
  • Workflow executes successfully.
  • API communication restored.
  • Automation resumes.

Key Takeaways

  • Operational visibility depends on every component of the monitoring pipeline functioning correctly, from endpoint telemetry collection through incident generation.
  • Monitoring failures should be investigated systematically by validating the virtualization platform, operating system, monitoring services, automation workflows, and downstream applications in sequence.
  • Recovery should prioritize service restoration and configuration validation before reverting virtual machines using snapshots or Tier 2 VZDump backups.
  • End-to-end testing is essential after every recovery activity to confirm that Sysmon telemetry, Wazuh detection, n8n automation, and Peppermint incident creation operate as an integrated system.
  • Comprehensive documentation and post-recovery validation ensure the monitoring platform is fully operational before returning the JCAG-LAB environment to production status.

Playbook Completion Outcome

Upon completion of this playbook: Capable of diagnosing and recovering monitoring infrastructure failures within the JCAG-LAB environment. This includes restoring Wazuh Manager services, recovering Wazuh Agent communication, validating Sysmon telemetry collection, resolving log ingestion problems, recovering Docker-based monitoring services, restoring n8n automation workflows, recovering Peppermint ITSM functionality, validating the complete end-to-end detection pipeline, and returning the enterprise monitoring platform to full operational status using structured operational recovery procedures.

LINUX ADMINISTRATION

Module 5: Linux Process & Service Troubleshooting

Overview

Processes and services are the foundation of Linux system operations. Every application, daemon, background task, and system function executes as one or more processes. When users report application failures, performance degradation, service outages, or server instability, Linux administrators must be able to identify, analyze, and resolve process-related issues efficiently.

This module focuses on process management, service administration, resource analysis, systemd troubleshooting, performance diagnostics, and common operational scenarios encountered in enterprise Linux environments.

Learning Objectives

  • Understand Linux process architecture.
  • Monitor running processes.
  • Identify resource-intensive processes.
  • Manage process lifecycles.
  • Troubleshoot failed services.
  • Analyze systemd service states.
  • Investigate startup failures.
  • Diagnose CPU, memory, and process-related issues.
  • Examine service dependencies.
  • Apply structured troubleshooting methodologies.

Understanding Linux Processes

A process is an executing instance of a program. Every process is assigned a unique Process ID (PID).

Examples:

  • nginx → Web server process
  • sshd → SSH daemon
  • systemd → System initialization process
  • wazuh-manager → Security monitoring service
  • mysql → Database service

Process Hierarchy

Linux processes operate in a parent-child structure.

Example process tree:

systemd

├─sshd
│ └─bash
├─cron
├─nginx
└─wazuh-manager

Process States

State Meaning
R Running
S Sleeping
D Uninterruptible sleep
T Stopped
Z Zombie
I Idle kernel process

Viewing Running Processes

  • ps aux → list all processes
  • ps -ef → detailed process listing
  • ps aux | grep nginx → filter specific process

Real-Time Process Monitoring

top and htop are used for live monitoring of CPU, memory, and process behavior.

  • top → built-in system monitor
  • htop → interactive enhanced monitor

Resource Usage Analysis

Sort processes by CPU:

ps aux --sort=-%cpu | head

Sort processes by memory:

ps aux --sort=-%mem | head

Check system memory:

free -h

Check load average:

uptime

Process Control

  • kill PID → terminate process
  • kill -9 PID → force kill
  • pkill nginx → kill by name
  • killall nginx → terminate all matching processes
  • nice -n 10 command → lower priority execution
  • renice PID → change priority of running process

Understanding systemd

systemd is the service manager responsible for starting, stopping, and managing services in modern Linux systems.

  • systemctl status service → check service state
  • systemctl start service → start service
  • systemctl stop service → stop service
  • systemctl restart service → restart service
  • systemctl reload service → reload configuration
  • systemctl enable service → start at boot
  • systemctl disable service → disable at boot

Starting Services

sudo systemctl start nginx

Stopping Services

sudo systemctl stop nginx

Restarting Services

sudo systemctl restart nginx

Reloading Configuration

sudo systemctl reload nginx

Enable Service at Boot

sudo systemctl enable nginx

Disable Service at Boot

sudo systemctl disable nginx

Investigating Failed Services

Check Status

systemctl status nginx

Example:

Active: failed

Review Service Logs

journalctl -u nginx

Recent Logs

journalctl -u nginx -n 50

View Errors Only

journalctl -p err

Service Dependency Analysis

Some services depend on other services or targets before they can start successfully.

Display Dependencies

systemctl list-dependencies nginx

Example:

network.target

syslog.target

Viewing Startup Services

systemctl list-unit-files --type=service

Boot Troubleshooting

Current Boot Logs

journalctl -b

Previous Boot Logs

journalctl -b -1

Boot Performance

systemd-analyze

Example:

Startup finished in 3.2s

Identify Slow Services

systemd-analyze blame

Example:

12.4s network-online.target

8.1s mysql.service

Monitoring Resource Consumption

CPU Statistics

mpstat

Install:

sudo apt install sysstat

Memory Statistics

vmstat

I/O Statistics

iostat

Combined Monitoring

sar

Identifying Zombie Processes

Display Zombies

ps aux | grep Z

Example:

STAT Z

Process Tree Investigation

pstree -p

Locate the parent process and determine why it has not cleaned up the child process.

Investigating Hung Processes

Processes in state D are in uninterruptible sleep and are typically waiting for hardware or storage operations to complete.

Check for Hung Processes

ps aux | grep D

Common causes:

  • Storage failures
  • NFS issues
  • Disk latency
  • Hardware problems

Application Port Verification

Check Listening Ports

ss -tulpn

Example:

tcp LISTEN 0.0.0.0:443

Verify Specific Service Port

ss -tulpn | grep nginx

File Handle Investigation

Display Open Files

lsof

Specific Process

lsof -p PID

Example:

lsof -p 4521

Port Usage

lsof -i :443

Common Troubleshooting Workflow

Service Not Running

  1. systemctl status service
  2. journalctl -u service
  3. ss -tulpn
  4. ps aux | grep service

High CPU Usage

  1. top
  2. ps aux --sort=-%cpu
  3. Investigate process before termination

High Memory Usage

  1. free -h
  2. ps aux --sort=-%mem
  3. vmstat

Application Unresponsive

  1. ps aux
  2. systemctl status
  3. journalctl -xe
  4. ss -tulpn

Key Takeaways

  • Every application and daemon running on Linux operates as a process.
  • ps, top, htop, and pstree are fundamental process analysis tools.
  • High CPU and memory consumption should be investigated systematically.
  • Services are managed through systemd and monitored using systemctl.
  • journalctl is the primary source for service troubleshooting.
  • Port verification and file handle analysis are critical for diagnosing application failures.
  • Boot performance and dependency analysis assist with startup troubleshooting.
  • A structured troubleshooting methodology significantly reduces mean time to resolution (MTTR).
  • Process and service troubleshooting skills are essential for Linux administrators, NOC engineers, SOC analysts, systems engineers, and infrastructure support personnel.

Module 6: Linux Log Analysis & Incident Investigation

Overview

Log analysis is one of the most important skills for Linux administrators, SOC analysts, NOC engineers, and incident responders. Nearly every security event, application failure, authentication issue, network problem, and system malfunction leaves evidence within system logs.

This module focuses on identifying log sources, analyzing system events, investigating security incidents, tracing user activity, correlating events across multiple log files, and developing a structured incident investigation methodology commonly used in enterprise environments.

Learning Objectives

  • Understand Linux logging architecture.
  • Identify common Linux log locations.
  • Analyze authentication events.
  • Investigate failed login attempts.
  • Monitor system activity.
  • Investigate service failures.
  • Perform timeline-based incident analysis.
  • Search large log datasets efficiently.
  • Correlate events from multiple log sources.
  • Conduct basic security investigations.

Understanding Linux Logging

Logs record system events generated by:

  • Operating system components
  • Authentication services
  • Applications
  • Network services
  • Security tools
  • Background daemons
  • User activity

Logs provide visibility into:

  • System failures
  • User logins
  • Privilege escalation
  • Service outages
  • Security incidents
  • Configuration changes

Understanding journalctl

View all journal entries:

journalctl

View recent entries:

journalctl -n 50

View live logs:

journalctl -f

View current boot logs:

journalctl -b

View previous boot:

journalctl -b -1

Filtering Logs

Filter by service:

journalctl -u nginx

Filter by SSH:

journalctl -u ssh

Filter by time:

journalctl --since today

Example:

journalctl --since "2026-06-01"

Specific time range:

journalctl --since "2026-06-20 10:00:00" --until "2026-06-20 11:00:00"

Log Severity Levels

Priority Description
emerg System unusable
alert Immediate action required
crit Critical condition
err Error
warning Warning
notice Significant event
info Informational
debug Debugging information

View errors only:

journalctl -p err

View warnings and above:

journalctl -p warning

Authentication Log Analysis

Authentication events are typically found in:

Ubuntu / Debian

/var/log/auth.log

Rocky Linux / AlmaLinux

/var/log/secure

Viewing Authentication Logs

Ubuntu

sudo cat /var/log/auth.log

Rocky Linux

sudo cat /var/log/secure

Search Failed Logins

Ubuntu

grep "Failed password" /var/log/auth.log

Rocky Linux

grep "Failed password" /var/log/secure

Example:

Failed password for invalid user admin from 192.168.1.100

Important fields:

Field Meaning
Failed password Login failure
invalid user Username does not exist
IP address Source host

Search Successful Logins

grep "Accepted password" /var/log/auth.log

Example:

Accepted password for admin from 10.10.100.50

Search SSH Activity

grep sshd /var/log/auth.log

Example:

sshd[1234]: Accepted password

User Activity Investigation

Display active sessions:

who

Display login history:

last

Example:

last

Output:

admin pts/0 10.10.100.50 Mon Jun 22

Failed Login History

lastb

Example:

lastb

Investigating Privilege Escalation

Search sudo activity:

grep sudo /var/log/auth.log

Example:

sudo: admin : TTY=pts/0 ; COMMAND=/usr/bin/systemctl restart nginx

Questions to ask:

  • Who executed the command?
  • When was it executed?
  • What command was run?

System Event Investigation

View system logs:

journalctl -xe

View recent events:

journalctl -n 100

View system errors:

journalctl -p err

Kernel Log Analysis

Kernel logs contain:

  • Driver events
  • Hardware failures
  • Memory issues
  • Disk problems

View kernel logs:

journalctl -k

Alternative:

dmesg

Recent kernel events:

dmesg | tail

Disk Failure Investigation

Search storage errors:

dmesg | grep error

Search disk failures:

dmesg | grep sda

Example:

I/O error on device sda

Common indicators:

  • I/O errors
  • Read failures
  • Write failures
  • Filesystem corruption

Memory Investigation

Search memory issues:

dmesg | grep memory

Search OOM events:

grep -i oom /var/log/syslog

Example:

Out of memory: Kill process 4521

Meaning:

Kernel terminated a process due to memory exhaustion.

Service Failure Investigation

Check service status:

systemctl status nginx

View service logs:

journalctl -u nginx

Recent logs:

journalctl -u nginx -n 50

Common causes:

  • Configuration errors
  • Missing files
  • Permission problems
  • Dependency failures

NGINX Log Investigation

Access log:

tail -f /var/log/nginx/access.log

Error log:

tail -f /var/log/nginx/error.log

View recent errors:

grep error /var/log/nginx/error.log

Apache Log Investigation

Access log:

tail -f /var/log/apache2/access.log

Error log:

tail -f /var/log/apache2/error.log

Searching Large Log Files

Search keyword:

grep ERROR logfile.log

Case insensitive:

grep -i error logfile.log

Count matches:

grep -c ERROR logfile.log

Display line numbers:

grep -n ERROR logfile.log

Advanced Log Searching

Search recursively:

grep -r "Failed password" /var/log

Search multiple patterns:

grep -E "ERROR|WARNING|CRITICAL" logfile.log

Invert results:

grep -v INFO logfile.log

Working with Large Logs

View large file:

less logfile.log

Search within less:

/ERROR

Next match:

n

Previous match:

N

Quit:

q

Real-Time Log Monitoring

Monitor file:

tail -f /var/log/syslog

Monitor authentication events:

tail -f /var/log/auth.log

Monitor NGINX:

tail -f /var/log/nginx/access.log

Log Correlation

Investigating Security Incidents

Common indicators:

  • Multiple failed logins
  • New user accounts
  • Unexpected sudo activity
  • Service modifications
  • Unknown outbound connections
  • Unauthorized file changes

Failed Login Investigation

Count failures:

grep "Failed password" /var/log/auth.log | wc -l

Top attacking IPs:

grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -nr

Example output:

145 192.168.1.50
90 10.10.10.25

User Account Investigation

View users:

cat /etc/passwd

Search recently created users:

grep home /etc/passwd

Review account changes:

grep useradd /var/log/auth.log

Sudo Abuse Investigation

Search sudo activity:

grep sudo /var/log/auth.log

Search specific user:

grep "admin" /var/log/auth.log

Questions:

  • Was sudo expected?
  • What command was executed?
  • Was execution authorized?

Network Activity Investigation

View Active Connections

ss -ant

View Listening Services

ss -tulpn

View Process Ownership

lsof -i

Example:

lsof -i :443

Detecting Unexpected Services

View listening ports:

ss -tulpn

Questions:

  • Is service expected?
  • Is port documented?
  • Is process legitimate?

File Integrity Investigation

Recently Modified Files

find /etc -mtime -1

Modified within 24 hours:

find / -type f -mtime -1

Questions:

  • What changed?
  • Who changed it?
  • Was change authorized?

Log Rotation

Linux rotates logs automatically.

Configuration:

/etc/logrotate.conf

View configuration:

cat /etc/logrotate.conf

View service configurations:

ls /etc/logrotate.d

Step 1 — Identify Event

Questions:

  • What happened?
  • Who reported it?
  • When did it occur?

Step 2 — Determine Scope

Questions:

  • Single host?
  • Multiple hosts?
  • Service impact?

Step 3 — Collect Evidence

Commands:

journalctl
grep
last
lastb
ss
lsof

Step 4 — Build Timeline

Document:

Time
↓
Event
↓
System response
↓
User impact

Step 5 — Determine Root Cause

Examples:

  • Misconfiguration
  • Authentication failure
  • Resource exhaustion
  • Hardware issue
  • Security incident

Step 6 — Remediation

Examples:

  • Restart service
  • Block IP address
  • Restore configuration
  • Patch vulnerability
  • Disable compromised account

Key Takeaways

  • Logs are the primary source of evidence during troubleshooting and incident response.
  • journalctl is the central tool for investigating modern Linux systems.
  • Authentication logs reveal login activity, brute-force attempts, and privilege escalation.
  • Service logs help identify application failures and startup issues.
  • Timeline reconstruction is one of the most effective investigation techniques.
  • Log correlation across multiple sources provides complete visibility into an incident.
  • Security investigations frequently begin with authentication, process, service, and network logs.
  • Structured investigation methodology improves accuracy and reduces incident response time.

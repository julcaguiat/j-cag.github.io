LINUX ADMINISTRATION

Module 7: Linux Server Recovery & Rescue Operations

Overview

Linux administrators are frequently required to recover systems that fail to boot, suffer filesystem corruption, experience service failures, lose network connectivity, or become inaccessible after configuration changes. Recovery operations are critical skills in enterprise environments where minimizing downtime and restoring business services are primary objectives.

This module focuses on system recovery procedures, rescue environments, bootloader troubleshooting, filesystem repair, password recovery, service restoration, and disaster recovery methodologies used in Linux infrastructure operations.

Learning Objectives

  • Understand Linux boot and recovery processes.
  • Access recovery and rescue environments.
  • Troubleshoot boot failures.
  • Repair damaged filesystems.
  • Recover root account access.
  • Restore failed services.
  • Recover network connectivity.
  • Repair bootloader issues.
  • Perform backup and recovery operations.
  • Follow structured disaster recovery procedures.

Understanding Linux Recovery Operations

Recovery operations involve restoring a system to a functional state after:

  • Failed system updates
  • Filesystem corruption
  • Bootloader failures
  • Service outages
  • Configuration mistakes
  • Lost administrator credentials
  • Hardware failures
  • Security incidents

Recovery objectives:

  1. Restore availability.
  2. Preserve data integrity.
  3. Minimize downtime.
  4. Identify root cause.
  5. Prevent recurrence.

Filesystem Recovery

Filesystems can become corrupted due to:

  • Power loss
  • Hardware failure
  • Storage controller issues
  • Improper shutdown

Detect Filesystem Problems

Review Logs

journalctl -p err

Review Kernel Messages

dmesg

Common indicators:

  • I/O error
  • Filesystem corruption
  • Read-only filesystem

Verify Filesystems

Display Mounted Devices

lsblk

Display Filesystem Usage

df -h

Unmount Filesystem

umount /dev/sdb1

Run Filesystem Check

Standard Check

fsck /dev/sdb1

Force Check

fsck -f /dev/sdb1

Automatically Repair

fsck -y /dev/sdb1

XFS Filesystem Recovery

Repair XFS Filesystem

xfs_repair /dev/sdb1

Example:

xfs_repair /dev/mapper/vg_data-lv_backup

  • Filesystem mounted read-only

mount

(ro)

Bootloader Recovery

GRUB corruption can prevent a Linux system from booting successfully.

Common symptoms:

  • grub rescue>
  • No bootable device
  • Operating system not found

Reinstall GRUB (Ubuntu)

Boot from live media.

Mount Root Partition

mount /dev/sda2 /mnt

Mount System Directories

mount --bind /dev /mnt/dev

mount –bind /proc /mnt/proc
mount –bind /sys /mnt/sys

Chroot into Installed System

chroot /mnt

Install GRUB

grub-install /dev/sda

Update Configuration

update-grub

Exit Chroot

exit

Reboot the system after completion.

Recovering Missing GRUB Configuration

Generate New Configuration

update-grub

Verify Configuration

cat /boot/grub/grub.cfg

Enterprise Recovery Workflow

Step 1 — Identify Failure

Questions:

  • What failed?
  • When did it fail?
  • What changed?

Step 2 — Assess Impact

Questions:

  • Single host?
  • Multiple systems?
  • Production impact?

Step 3 — Preserve Evidence

Collect:

  • journalctl output
  • dmesg output
  • systemctl status results
  • Network configuration
  • Screenshots

Step 4 — Perform Recovery

Examples:

  • Filesystem repair
  • Service restart
  • Password reset
  • Network restoration
  • Bootloader repair

Step 5 — Validate Recovery

Verify:

  • Service status
  • Application functionality
  • Network connectivity
  • User access
  • Monitoring systems

Step 6 — Document Findings

Record:

  • Root cause
  • Actions performed
  • Recovery timeline
  • Lessons learned

Key Takeaways

  • Recovery operations are focused on restoring system availability while preserving data integrity.
  • Understanding the Linux boot process is critical for diagnosing startup failures.
  • Filesystem corruption is commonly repaired using fsck or xfs_repair.
  • GRUB recovery procedures are essential for resolving bootloader failures.
  • Service recovery requires log analysis, dependency verification, and configuration validation.
  • Network recovery should follow a structured approach involving interfaces, routing, and DNS.
  • Backups are a critical component of disaster recovery planning.
  • Thorough documentation improves future recovery efforts and supports operational maturity.
  • Effective recovery operations significantly reduce downtime and improve service reliability.
